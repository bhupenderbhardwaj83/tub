01

History & Evolution

React started as a purely client-side library. This led to the "loading spinner hell" and poor SEO. Vercel (then Zeit) created Next.js to solve Server-Side Rendering (SSR) easily.

Classic SPA

Empty HTML shell, JS loads, then fetches data. Slow initial paint. Bad for SEO.

Pages Router (Next 12)

Introduced `getServerSideProps` & `getStaticProps`. Route based on file system.

App Router (Modern)

React Server Components (RSC) default. Fetch inside components. Streaming. Suspense.

02

Underlying Architecture

Next.js is not just React; it's a full-stack framework with a hybrid rendering engine.

  • Server-First: By default, everything is a Server Component. It stays on the server, reducing bundle size.
  • Edge Runtime: Middleware acts as the gatekeeper, running on Vercel's Edge (a lightweight version of Node.js) for lowest latency.
  • Hydration: The "Dry HTML" is sent first, then "Hydrated" with interactivity (React events) only where needed (`'use client'`).

04

Routing System

Dynamic Routes

app/blog/[slug]/page.tsx

Matches /blog/a, /blog/b. Access via params.slug.

Catch-All Routes

app/docs/[...slug]/page.tsx

Matches /docs/a, /docs/a/b. Array of segments.

Route Groups

app/(marketing)/page.tsx

Organize code without affecting URL structure. useful for multiple root layouts.

Private Folders

app/_utils/helper.ts

Folders starting with _ are ignored by the router.

05

Rendering Models (The Core Decision)

Pattern Acronym When to use
Server Components RSC Default. Fetching data, accessing backend resources, sensitive logic.
Client Components CSR Interactivity, onClick, useState, browser APIs (localStorage).
Static Generation SSG Marketing pages, Blogs. content doesn't change often. Extremely fast.
Incremental Regen ISR E-commerce products. Update static pages after a timeout without rebuilding app.
06

Component Patterns

Architecture Golden Rule: Move Clients Components down the tree over "Leaf" nodes. Keep layout and data fetching in Server Components. 
// app/page.tsx (Server Component)
import ClientCounter from './ClientCounter';

export default async function Page() {
  // Fetch happens on server
  const data = await db.getStuff();

  return (
    <div>
      <h1>{data.title}</h1>
      {/* Interactivity isolated here */}
      <ClientCounter initialCount={data.count} />
    </div>
  );
}
07

State Management

URL as State

In Next.js, the URL is the ultimate source of truth. Search queries, filters, and pagination should be URL params, not React state. This makes pages shareable.

  • Avoid `useState` for filters.
  • Use `useSearchParams` and `useRouter`.
  • Enable bookmarking of deep application states.

Server State

Don't use Redux/Zustand for server data. Use React Query (TanStack Query) or simply rely on Next.js Cache + Revalidation.

08

Data Fetching & Async

Next.js extends the native `fetch` API to allow granular caching and revalidation.

// Cached (Static - like getStaticProps)
fetch('...', { force_cache: 'force-cache' });

// Dynamic (SSR - like getServerSideProps)
fetch('...', { cache: 'no-store' });

// ISR (Revalidate every hour)
fetch('...', { next: { revalidate: 3600 } });

Request Memoization

If you call the same fetch in Layout, Page, and a Component, Next.js automatically dedupes it. You don't need to pass props down 10 levels.

09

Forms & Server Actions

Server Actions allow you to write functions that execute on the server but are called from the client forms. No API routes needed.

// actions.ts
'use server'

export async function createUser(formData: FormData) {
    const rawData = Object.fromEntries(formData);
    await db.user.create(rawData);
    revalidatePath('/users');
    redirect('/users');
}

// component.tsx
return (
    <form action={createUser}>
       <input name="username" />
       <button type="submit">Save</button>
    </form>
)

Search & Indexing

Implementing a global search (Cmd+K) involves:

  1. Indexing: On build time (SSG) or periodic jobs, scan your content.
  2. Storage: Store simplified JSON index or use Algolia/Elasticsearch.
  3. UI: A Dialog modal (accessible) that fetches results on input.
12

Dashboards & Graphs

Libraries like Recharts or Victory are heavy. Render the shell on Server, and lazy-load the heavy Chart components.

import dynamic from 'next/dynamic';

const HeavyChart = dynamic(() => import('./HeavyChart'), {
  loading: () => <p>Loading Graph...</p>,
  ssr: false // Often charts need window object
});
13

Authentication Architecture

For Enterprise apps, Auth logic belongs in the Backend (FastAPI), not the Frontend.

The Pattern

Next.js acts as a proxy. It receives credentials, sends them to FastAPI. FastAPI returns an HttpOnly, Secure Cookie. Next.js forwards this cookie on every subsequent request.

NextAuth.js (Auth.js)

Great for side-projects or simple social login. For hard enterprise requirements (RBAC, MFA, Custom Logic), custom backend integration is preferred.

14

FastAPI Integration (Python Backend)

The perfect stack: Next.js (Frontend) + FastAPI (Backend).

API Route Wrapper Pattern

Don't expose FastAPI directly to the browser. Use Next.js API Routes (Route Handlers) as a secure gateway.

// app/api/login/route.ts
import { cookies } from 'next/headers';

export async function POST(request: Request) {
  const body = await request.json();
  
  // Call FastAPI
  const res = await fetch('http://backend:8000/token', {
      method: 'POST',
      body: JSON.stringify(body)
  });
  
  const data = await res.json();
  
  // Set backend cookie on Next.js/Browser
  cookies().set('session', data.access_token, { httpOnly: true });
  
  return Response.json({ success: true });
}

Environment & Performance

Env: Use NEXT_PUBLIC_ prefix only for browser variables. Secrets must stay in `.env.local` and never be prefixed.

Images: `next/image` is mandatory for LCP optimization. It handles lazy loading and resizing automatically.

Mobile Strategy: React Native

The "Solito" Pattern

You can share up to 90% of code between Next.js (Web) and React Native (Mobile). Solito is the missing piece that unifies navigation.

  • Shared: UI Components, Business Logic, API Hooks (TanStack Query), Zod Schemas.
  • Web only: Next.js Pages, SEO tags.
  • Mobile only: Expo Router, Native Modules (Camera, Gyroscope).
folder structure
apps/
  next/     # Next.js app
  expo/     # React Native app
packages/
  ui/       # Shared UI (Tamagui/NativeWind)
  app/      # Shared Screens & Logic
