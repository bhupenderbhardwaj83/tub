What is Docker?

Docker is an open platform for developing, shipping, and running applications. It enables you to separate your applications from your infrastructure so you can deliver software quickly.

The Problem It Solves

"It works on my machine" is a classic developer cliché. Environments differ (OS, dependencies, versions). Docker packages the code and its dependencies into a standard unit called a container.

VMs vs Containers

  • VMs: Heavy. Include full OS guest. Slow boot. GBs in size.
  • Containers: Lightweight. Share Host OS Kernel. Instant startup. MBs in size.

Core Concepts

🖼️

Image

Read-only template with instructions to create a container. Like a blueprint or a class in OOP.

📦

Container

Runnable instance of an image. Isolated execution environment. Like an object instance in OOP.

🏯

Registry

Library where images are stored (e.g., Docker Hub). Like git remote repositories.

Terminal
$ docker run -d -p 80:80 nginx
# 1. Docker client contacts Docker Daemon
# 2. Daemon pulls 'nginx' image from Registry
# 3. Creates new container from image
# 4. Maps host port 80 to container port 80

Architecture

Client-Server Model

Docker uses a client-server architecture. The Docker client talks to the Docker daemon, which does the heavy lifting of building, running, and distributing your containers.

Client (CLI) Sends commands via REST API
➡
Docker Daemon (dockerd) Manages objects (images, containers)

Under the Hood

Namespaces

Provide isolation. Each container has its own view of the system (PID, NET, MNT, etc.).

Control Groups (cgroups)

Manage resources. Limit how much CPU or Memory a container can use.

Docker Images

Layers & Caching

Images are built in layers. Each instruction in a Dockerfile creates a layer. Layers are cached to speed up builds.

FROM ubuntu:22.04 Base Layer (Cached)
RUN apt-get update Layer 1 (Cached)
COPY . . Layer 2 (Changed)
CMD ["npm", "start"] Layer 3 (New)

Tip: Put frequently changing instructions (like COPYing source code) at the bottom to maximize cache hits.

6

Dockerfile

Instruction Description Best Practice
FROM Base image Use specific tags (node:18-alpine) not latest.
WORKDIR Sets working directory Always set this to keep things organized (e.g., /app).
COPY Copies files Prefer COPY over ADD unless extracting tarballs.
RUN Executes commands Chain commands with && to reduce layers.
CMD Default command Use JSON array format: CMD ["npm", "start"].

Multi-Stage Builds

Drastically reduce image size by separating build tools from the runtime.

# Stage 1: Build
FROM node:18 AS builder
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build

# Stage 2: Run
FROM node:18-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
CMD ["node", "dist/main.js"]

Networking

Network Drivers

Bridge (Default) Private internal network. Containers talk via IP. NAT to outside.
Host Remove isolation. Container shares host IP & ports directly. Fast but risky.
None No networking. Complete isolation. Good for batch jobs needing security.

Volumes & Storage

Warning: Containers are ephemeral. If you delete a container, its data is gone forever unless you use volumes.

Bind Mounts

Maps a specific path on your Host to the Container.

  • Performance depends on Host FS
  • Great for development (live reload)
-v /home/user/code:/app

Named Volumes

Recommended

Managed by Docker. Stored in Docker's area.

  • Higher performance
  • Easier to backup/migrate
-v my-db-data:/var/lib/mysql

Docker Compose

Define and run multi-container applications. Instead of running 5 separate `docker run` commands, write one YAML file. 

version: '3.8'

services:
  web:
    build: .
    ports:
      - "3000:3000"
    depends_on:
      - db
    environment:
      - DB_HOST=db

  db:
    image: postgres:15
    volumes:
      - db-data:/var/lib/postgresql/data
    environment:
      - POSTGRES_PASSWORD=secret

volumes:
  db-data:

Security Best Practices

Don't run as Root

If the container is compromised, the attacker has root on host. Create a `USER` in Dockerfile.

Use Minimal Base Images

Alpine or Distroless images have fewer vulnerabilities and smaller surface area.

Scan Images

Use tools like generic or Synk to scan for CVEs in your images.

Secrets Management

Never bake secrets into the Image. Use ENV vars or Docker Secrets.

Practical Labs

Lab 1: Static Website with NGINX

Host a simple HTML page using the official Nginx image.

# 1. Create index.html
echo "<h1>Hello Docker</h1>" > index.html

# 2. Run container (bind mount current dir to nginx html dir)
docker run -d -p 8080:80 -v $(pwd):/usr/share/nginx/html --name my-site nginx

Expected: Visit http://localhost:8080 to see "Hello Docker".

Lab 2: Python / Node API

Containerise a simple Python Flask app.

app.py 
from flask import Flask
app = Flask(__name__)

@app.route('/')
def hello():
    return "Hello from Container!"

if __name__ == '__main__':
    app.run(host='0.0.0.0')
Dockerfile 
FROM python:3.9-slim
WORKDIR /app
COPY . .
RUN pip install flask
CMD ["python", "app.py"]
docker build -t my-api .
docker run -p 5000:5000 my-api

Lab 3: Full Stack (Compose)

Wordpress + MySQL using Docker Compose.

services:
  db:
    image: mysql:5.7
    volumes:
      - db_data:/var/lib/mysql
    environment:
      MYSQL_ROOT_PASSWORD: example
      MYSQL_DATABASE: wordpress

  wordpress:
    image: wordpress:latest
    depends_on:
      - db
    ports:
      - "8000:80"
    environment:
      WORDPRESS_DB_HOST: db:3306
      WORDPRESS_DB_USER: root
      WORDPRESS_DB_PASSWORD: example

volumes:
  db_data:
docker-compose up -d

Visit http://localhost:8000 to install Wordpress.

Lab 4: Persistent Database

Prove data persistence by removing the container.

  1. Start Postgres with a named volume:
    docker run -d --name pg -v pgdata:/var/lib/postgresql/data -e POSTGRES_PASSWORD=mysecret postgres
  2. Exec in and create a table/data.
  3. Stop and Remove container: docker rm -f pg
  4. Start a new container with same volume mapping.
  5. Verify data is still there.

Lab 5: Reverse Proxy (Traefik)

Advanced: Use Traefik to route traffic to containers dynamically.

services:
  traefik:
    image: traefik:v2.9
    command: --api.insecure=true --providers.docker
    ports:
      - "80:80"
      - "8080:8080"
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock

  whoami:
    image: traefik/whoami
    labels:
      - "traefik.http.routers.whoami.rule=Host(`whoami.localhost`)"

Run this, then visit http://whoami.localhost (if DNS resolves) or check Traefik dashboard at http://localhost:8080.
