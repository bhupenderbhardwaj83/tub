A.1 String Methods

Complete String Manipulation Toolkit

Python strings have 40+ built-in methods for common text operations. These methods are highly optimized C implementations, much faster than manual loops. They're immutable, so all methods return new strings.

💡 What Is It?

Built-in functions attached to every string object that manipulate, search, validate, or transform text without external libraries.

🔧 Why It Exists?

Text processing is universal. Without these methods, you'd write complex loops for simple tasks like "remove whitespace" or "find substring".

⚙️ How It Works?

Methods are implemented in C, operating directly on string bytes in memory. They're immutable - original string unchanged, new one returned.

🎯 Where Used?

Data cleaning, parsing logs, API response handling, form validation, file processing, web scraping, text analysis.

✅ When To Use?

Always prefer built-in methods over manual loops. Use for any string manipulation - they're optimized and battle-tested.

⚠️ When To Avoid?

For complex parsing (use regex), unicode normalization (use unicodedata), or heavy text processing (use libraries like nltk).

Simple Example

# Common string methods
text = "  Hello, World!  "

# Cleaning
print(text.strip())         # "Hello, World!"
print(text.lower())         # "  hello, world!  "
print(text.replace("World", "Python"))  # "  Hello, Python!  "

# Searching
print(text.find("World"))   # 9 (index)
print("World" in text)      # True

# Splitting / Joining
words = "one,two,three".split(",")
print(words)                # ['one', 'two', 'three']
print("-".join(words))       # "one-two-three"

Production Example: Log Parser

# Real-world: Parse web server logs
def parse_log_line(line):
    """
    Parse: 192.168.1.1 - - [10/Oct/2024:13:55:36] "GET /api/users HTTP/1.1" 200
    """
    line = line.strip()  # Remove whitespace
    
    if not line or line.startswith("#"):
        return None  # Skip empty/comments
    
    # Extract IP (before first space)
    ip = line.split()[0]
    
    # Extract HTTP method
    if '"' in line:
        request = line.split('"')[1]
        method = request.split()[0]
        path = request.split()[1]
    else:
        return None
    
    # Extract status code (last number)
    parts = line.split()
    status = parts[-1] if parts[-1].isdigit() else "000"
    
    return {
        'ip': ip,
        'method': method,
        'path': path,
        'status': int(status)
    }

# Usage
log = '192.168.1.1 - - [10/Oct/2024] "GET /api/users HTTP/1.1" 200'
parsed = parse_log_line(log)
print(parsed)
# {'ip': '192.168.1.1', 'method': 'GET', 'path': '/api/users', 'status': 200}

❌ Common Mistakes

1. Forgetting strings are immutable:

text.strip() doesn't modify text - use text = text.strip()

2. Using + in loops:

Never: result = "" then result += word in loop. Use "".join(words) instead.

3. Not handling edge cases:

text.split(",") on empty string returns [""] not []

✅ Quick Revision Checklist

  • strip(), lstrip(), rstrip() - Remove whitespace
  • lower(), upper(), title(), capitalize() - Case conversion
  • split(), join() - String ↔ List conversion
  • replace(), find(), index() - Search and replace
  • startswith(), endswith() - Prefix/suffix checks
  • isdigit(), isalpha(), isalnum() - Validation
  • format(), f-strings - Modern string formatting

A.2 List Methods

Mutable Sequence Operations Toolkit

Python lists come with 11 built-in methods for adding, removing, sorting, and manipulating elements. Unlike strings, lists are mutable - methods modify the list in-place.

💡 What Is It?

Built-in functions for list manipulation: append(), extend(), insert(), remove(), pop(), clear(), index(), count(), sort(), reverse(), copy().

🔧 Why It Exists?

Dynamic collections need efficient operations. Methods provide optimized alternatives to manual loops for common tasks.

⚙️ How It Works?

Most methods modify in-place (return None), operating on underlying array. Methods like count() and index() are read-only.

🎯 Where Used?

Data pipelines, stacks/queues, filtering, sorting, batch operations, dynamic collections.

✅ When To Use?

Always prefer built-in methods. Use append() for single items, extend() for merging, sort() for ordering, pop() for stacks.

⚠️ When To Avoid?

Fixed-size data (use tuples), frequent lookups (use sets/dicts), immutability needed. Avoid sort() in tight loops.

Simple Example

fruits = ['apple', 'banana']
fruits.append('orange')
fruits.extend(['grape', 'mango'])
fruits.insert(1, 'kiwi')
print(fruits)

numbers = [3, 1, 4, 1, 5]
numbers.sort()
print(numbers)
['apple', 'kiwi', 'banana', 'orange', 'grape', 'mango'] [1, 1, 3, 4, 5]

❌ Common Mistakes

1. Confusing append() with extend(): append([1,2]) adds nested list, extend([1,2]) adds two items

2. Expecting return value: list.sort() returns None, modifies in-place

✅ Quick Revision

  • append(x) - Add item to end
  • extend(iter) - Add all items from iterable
  • insert(i, x) - Insert at index
  • remove(x) - Remove first occurrence
  • pop(i) - Remove and return at index
  • sort() - Sort in-place
  • reverse() - Reverse in-place

A.3 Tuple Methods

Immutable Sequence Operations

Tuples have only 2 built-in methods: count() and index(). This minimal API reflects their immutable nature - you can't add, remove, or modify elements. Tuples are used for fixed collections and hashable sequences.

💡 What Is It?

count(x) returns occurrences of x, index(x) returns position of first x. That's it - tuples are intentionally minimal.

🔧 Why It Exists?

Tuples guarantee immutability. Limited methods prevent modification, ensuring data integrity and enabling use as dict keys.

⚙️ How It Works?

Methods perform linear search through tuple. No modification operations exist - attempting them raises AttributeError.

🎯 Where Used?

Function return values (multiple), dict keys, coordinates (x,y), RGB colors, database records, configuration data.

✅ When To Use?

Use tuples when data shouldn't change: constants, function returns, dict keys. Faster and more memory-efficient than lists.

⚠️ When To Avoid?

Don't use when you need to modify data frequently. Lists are better for dynamic collections. Can't sort or reverse tuples.

Simple Example

# Tuple methods
coordinates = (10, 20, 10, 30, 10)

# count() - how many times value appears
print("Count of 10:", coordinates.count(10))

# index() - position of first occurrence
print("Index of 20:", coordinates.index(20))

# Common use: unpacking
point = (5, 15)
x, y = point
print(f"X: {x}, Y: {y}")

# As dict key (immutable)
locations = {
    (0, 0): "origin",
    (1, 0): "right"
}
print(locations[(0, 0)])
Count of 10: 3 Index of 20: 1 X: 5, Y: 15 origin

❌ Common Mistakes

1. Trying to modify tuples: tuple.append(x) raises AttributeError - tuples are immutable

2. Single-element syntax: (5) is not a tuple, (5,) is. Need trailing comma.

3. Using when list needed: Can't sort/modify tuples. Convert to list first: list(tuple)

✅ Quick Revision

  • count(x) - Return number of occurrences of x
  • index(x) - Return index of first x (raises ValueError if not found)
  • Tuples are immutable - no add/remove/sort methods
  • Use for: constants, function returns, dict keys
  • Faster and more memory-efficient than lists
  • Single element syntax: (x,) not (x)

A.4 Set Methods

Unordered Collection with Mathematical Operations

Sets provide 15+ methods for mathematical set operations (union, intersection, difference), membership testing, and uniqueness. They're unordered, mutable, and store only unique elements - perfect for deduplication and membership testing with O(1) average time complexity.

💡 What Is It?

Methods for set operations: add(), remove(), discard(), union(), intersection(), difference(), symmetric_difference(), plus subset/superset tests.

🔧 Why It Exists?

Fast membership testing (O(1)), automatic deduplication, mathematical set operations. Hash-based implementation makes lookups extremely fast.

⚙️ How It Works?

Hash table stores elements. Each element must be hashable (immutable). Operations use hashing for O(1) average access.

🎯 Where Used?

Removing duplicates, membership testing, finding common/unique items, tag systems, permissions, category filtering, data comparison.

✅ When To Use?

Use for: uniqueness, fast lookups, mathematical operations, filtering. Much faster than lists for 'in' checks on large collections.

⚠️ When To Avoid?

Don't use when: order matters, duplicates needed, elements aren't hashable (lists/dicts), need indexing, or frequent iteration.

Simple Example

# Creating and modifying sets
tags = {'python', 'coding', 'tutorial'}
tags.add('beginner')
tags.remove('coding')  # Raises KeyError if not found
tags.discard('advanced')  # Safe - no error if not found

# Set operations
skills_a = {'python', 'java', 'sql'}
skills_b = {'python', 'javascript', 'sql'}

print("Union:", skills_a | skills_b)
print("Intersection:", skills_a & skills_b)
print("Difference:", skills_a - skills_b)
print("Symmetric diff:", skills_a ^ skills_b)

# Deduplication
numbers = [1, 2, 2, 3, 3, 3]
unique = list(set(numbers))
print("Unique:", unique)
Union: {'python', 'java', 'sql', 'javascript'} Intersection: {'python', 'sql'} Difference: {'java'} Symmetric diff: {'java', 'javascript'} Unique: [1, 2, 3]

❌ Common Mistakes

1. Using unhashable types: set([1, [2, 3]]) fails - lists aren't hashable. Use tuples instead.

2. Expecting order: Sets are unordered. Can't index or slice: set[0] raises TypeError.

3. remove() vs discard(): remove() crashes if item missing, discard() is safe.

✅ Quick Revision

  • add(x) - Add element
  • remove(x) - Remove (error if missing)
  • discard(x) - Remove (safe)
  • union() or | - All elements from both
  • intersection() or & - Common elements
  • difference() or - - In first, not second
  • symmetric_difference() or ^ - In either, not both

A.5 Dictionary Methods

Key-Value Mapping Operations

Dictionaries provide 20+ methods for manipulating key-value pairs. Core methods include get(), keys(), values(), items(), update(), pop(), setdefault(). Dicts are ordered (Python 3.7+), mutable, and provide O(1) average lookup time.

💡 What Is It?

Methods for dict operations: get(), keys(), values(), items(), update(), pop(), popitem(), setdefault(), clear(), copy(), fromkeys().

🔧 Why It Exists?

Key-value storage is fundamental. Methods provide safe access (get), iteration (items), merging (update), and atomic operations.

⚙️ How It Works?

Hash table maps keys to values. Keys must be hashable. Methods operate on this mapping with O(1) average complexity.

🎯 Where Used?

Configuration, caching, counting, grouping, lookups, JSON data, API responses, databases, state management.

✅ When To Use?

Use for: key-value associations, fast lookups, counting, configuration. get() for safe access, setdefault() for defaults.

⚠️ When To Avoid?

Don't use when: order by value needed (use sorted), multiple values per key (use lists), or keys aren't hashable.

Simple Example

# Safe access with get()
user = {'name': 'Alice', 'age': 30}
print(user.get('email', 'No email'))  # Returns default

# Iteration methods
print("Keys:", list(user.keys()))
print("Values:", list(user.values()))
print("Items:", list(user.items()))

# Update and merge
user.update({'city': 'NYC', 'age': 31})
print(user)

# setdefault - get or create
counts = {}
for letter in 'hello':
    counts.setdefault(letter, 0)
    counts[letter] += 1
print(counts)

# Pop with default
city = user.pop('city', 'Unknown')
print(f"Removed: {city}")
No email Keys: ['name', 'age'] Values: ['Alice', 30] Items: [('name', 'Alice'), ('age', 30)] {'name': 'Alice', 'age': 31, 'city': 'NYC'} {'h': 1, 'e': 1, 'l': 2, 'o': 1} Removed: NYC

❌ Common Mistakes

1. KeyError with direct access: dict[key] crashes if missing. Use dict.get(key, default) instead.

2. Modifying during iteration: Can't add/remove keys while iterating. Make a copy first: dict.copy().

3. Mutable default values: Don't use mutable defaults with setdefault() or get() - they're shared.

✅ Quick Revision

  • get(key, default) - Safe access with default
  • keys() - View of all keys
  • values() - View of all values
  • items() - View of (key, value) pairs
  • update(other) - Merge dictionaries
  • pop(key, default) - Remove and return
  • setdefault(key, default) - Get or insert default

B.1 Context Managers (with)

Automatic Resource Management

Context managers handle setup and teardown automatically using 'with' statement. They guarantee cleanup even if exceptions occur, preventing resource leaks (file handles, locks, connections). Implements __enter__() and __exit__() methods.

💡 What Is It?

Objects that manage resources using 'with' statement. Automatically call setup (__enter__) and cleanup (__exit__) code.

🔧 Why It Exists?

Manual cleanup is error-prone. Exceptions can skip cleanup code. Context managers guarantee cleanup happens, preventing leaks.

⚙️ How It Works?

__enter__() runs at 'with' start, __exit__() at end (even on exception). Commonly used with files, locks, database connections.

🎯 Where Used?

File I/O, database connections, thread locks, network sockets, temporary files, transaction management.

✅ When To Use?

Always use 'with' for files. Use for any resource needing cleanup: connections, locks, temp files. Prevents resource leaks.

⚠️ When To Avoid?

Not needed for simple variables or calculations. Overkill for resources that don't need cleanup. Adds nesting/indentation.

Simple Example

# ❌ Without context manager - manual cleanup
f = open('file.txt', 'r')
data = f.read()
f.close()  # Easy to forget!

# ✅ With context manager - automatic cleanup
with open('file.txt', 'r') as f:
    data = f.read()
# File automatically closed here, even if exception occurs

# Multiple resources
with open('input.txt') as fin, open('output.txt', 'w') as fout:
    fout.write(fin.read().upper())

✅ Quick Revision

  • Always use with for file operations
  • Guarantees cleanup even on exceptions
  • Can manage multiple resources in one statement
  • Works with files, locks, connections, custom classes

B.2 Reading Files

File Input Operations

Python provides read(), readline(), readlines() for file input. read() loads entire file, readline() one line, readlines() all lines as list. For large files, iterate directly over file object for memory efficiency.

💡 What Is It?

Methods to read file contents: read() (all), readline() (one line), readlines() (list of lines), or iterate line-by-line.

🔧 Why It Exists?

Different use cases need different approaches. Small files: read all. Large files: iterate. Specific lines: readline().

⚙️ How It Works?

File pointer tracks position. read() moves to end, readline() to next newline. Iteration uses buffer for efficiency.

🎯 Where Used?

Config files, logs, CSV processing, data import, text analysis, parsing documents.

✅ When To Use?

read() for small files. Iterate for large files. readline() for processing line-by-line with custom logic.

⚠️ When To Avoid?

Don't use read() on huge files - loads all to memory. Don't use readlines() for gigabyte files. Use iteration.

Simple Example

# Read entire file
with open('data.txt') as f:
    content = f.read()
    print(content)

# Read line by line (memory efficient for large files)
with open('log.txt') as f:
    for line in f:
        if 'ERROR' in line:
            print(line.strip())

# Read all lines as list
with open('names.txt') as f:
    lines = f.readlines()
    print(len(lines), "lines")

✅ Quick Revision

  • read() - Entire file as string
  • readline() - One line at a time
  • readlines() - All lines as list
  • Iteration - Most memory-efficient for large files

B.3 Writing Files

File Output Operations

write() and writelines() handle file output. write() writes string, writelines() writes list of strings. Must add newlines manually with write(). Use flush() to force immediate write to disk.

💡 What Is It?

write(string) writes one string, writelines(list) writes multiple. Both return bytes written. No automatic newlines.

🔧 Why It Exists?

Programs need to persist data. Writing to files stores results, logs, exports, reports, and configurations.

⚙️ How It Works?

Data buffered in memory first, then written to disk. flush() forces immediate write. close() auto-flushes.

🎯 Where Used?

Logging, reports, data export, config generation, cache files, saving state.

✅ When To Use?

write() for single strings, writelines() for lists. Always use 'with' to ensure data is saved.

⚠️ When To Avoid?

Don't write sensitive data without encryption. Don't forget newlines. Avoid many small writes - buffer them.

Simple Example

# Write single string
with open('output.txt', 'w') as f:
    f.write("Hello, World!\n")
    f.write("Second line\n")

# Write multiple lines
lines = ["Line 1\n", "Line 2\n", "Line 3\n"]
with open('multiple.txt', 'w') as f:
    f.writelines(lines)

# Append mode
with open('log.txt', 'a') as f:
    f.write("New log entry\n")

✅ Quick Revision

  • write(str) - Write single string
  • writelines(list) - Write list of strings
  • Must add '\n' manually for newlines
  • Use 'w' mode (overwrites) or 'a' mode (appends)

B.4 File Modes

Open Modes and Binary Operations

File modes control read/write behavior: 'r' (read), 'w' (write/overwrite), 'a' (append), 'x' (exclusive create). Add 'b' for binary mode, '+' for read/write. Default is text mode with encoding.

💡 What Is It?

Mode string controls file behavior: r (read), w (write), a (append), x (create), b (binary), + (update).

🔧 Why It Exists?

Different operations need different permissions. Prevents accidental overwrites. Binary mode for non-text files.

⚙️ How It Works?

OS opens file with specified permissions. 'w' truncates, 'a' seeks to end, 'b' disables encoding.

🎯 Where Used?

Text files (r/w/a), images (rb/wb), logs (a), safe creates (x), read-modify-write (r+).

✅ When To Use?

'r' for reading, 'w' to create/overwrite, 'a' for logs, 'rb'/'wb' for binary files (images, PDFs).

⚠️ When To Avoid?

'w' destroys existing data - use 'r+' if you need to preserve. Don't mix text/binary modes.

Simple Example

# Text modes
with open('file.txt', 'r') as f:  # Read (must exist)
    data = f.read()

with open('file.txt', 'w') as f:  # Write (overwrites!)
    f.write("New content")

with open('log.txt', 'a') as f:  # Append
    f.write("Log entry\n")

# Binary modes (for images, etc)
with open('image.jpg', 'rb') as f:
    bytes_data = f.read()

with open('copy.jpg', 'wb') as f:
    f.write(bytes_data)

✅ Quick Revision

  • 'r' - Read (default, must exist)
  • 'w' - Write (creates/truncates)
  • 'a' - Append (creates if missing)
  • 'x' - Exclusive create (fails if exists)
  • 'b' - Binary mode (no encoding)
  • '+' - Read and write

B.5 JSON & CSV

Structured Data Formats

JSON and CSV are standard data formats. json module handles JSON (loads/dumps, load/dump). csv module reads/writes CSV with DictReader/DictWriter for dict-based access.

💡 What Is It?

json: loads() parses string, load() reads file, dumps() creates string, dump() writes file. csv: reader/writer, DictReader/DictWriter.

🔧 Why It Exists?

Structured data needs standard formats. JSON for nested data/APIs. CSV for tabular data/spreadsheets.

⚙️ How It Works?

JSON converts Python types ↔ JSON text. CSV reads/writes rows, handles quotes/delimiters, supports headers.

🎯 Where Used?

APIs, configs, data exchange. CSV for spreadsheets, exports, data science, bulk imports.

✅ When To Use?

JSON for nested/complex data. CSV for flat tables. DictReader for header-based access.

⚠️ When To Avoid?

JSON not for binary data. CSV struggles with nested data. Use pandas for large CSV processing.

Simple Example

import json
import csv

# JSON - Save and load
data = {'name': 'Alice', 'score': 95}
with open('data.json', 'w') as f:
    json.dump(data, f, indent=2)

with open('data.json') as f:
    loaded = json.load(f)

# CSV - Write and read
rows = [
    ['Name', 'Age'],
    ['Alice', 30],
    ['Bob', 25]
]
with open('people.csv', 'w', newline='') as f:
    writer = csv.writer(f)
    writer.writerows(rows)

with open('people.csv') as f:
    reader = csv.DictReader(f)
    for row in reader:
        print(row['Name'], row['Age'])

✅ Quick Revision

  • json.load(f) - Parse JSON file
  • json.dump(data, f) - Write JSON file
  • csv.reader(f) - Read CSV rows
  • csv.DictReader(f) - Read as dicts with headers
  • csv.writer(f) - Write CSV rows

C.1 GET Requests

HTTP GET Method for Data Retrieval

requests.get() fetches data from URLs. Returns Response object with status_code, text, json(), headers. GET is idempotent (safe to repeat), used for retrieving data without side effects. Most common HTTP method.

💡 What Is It?

requests.get(url) sends HTTP GET request, returns Response with status, headers, body. Can include params and headers.

🔧 Why It Exists?

Web APIs use HTTP protocol. GET retrieves data from servers: APIs, web pages, images, JSON data. Standard web communication.

⚙️ How It Works?

Creates HTTP GET request with URL, sends to server, waits for response. Response contains status code, headers, body.

🎯 Where Used?

API consumption, web scraping, downloading files, fetching JSON data, accessing REST endpoints, health checks.

✅ When To Use?

Use GET for reading/retrieving data. Safe for caching, bookmarking, sharing. No sensitive data in URL.

⚠️ When To Avoid?

Don't use GET for: submitting forms, sending sensitive data, modifying server state. Use POST instead.

Simple Example

import requests

# Basic GET request
response = requests.get('https://api.github.com')
print("Status:", response.status_code)
print("Content-Type:", response.headers['Content-Type'])

# GET with parameters
params = {'q': 'python', 'sort': 'stars'}
response = requests.get('https://api.github.com/search/repositories', 
                           params=params)

# Parse JSON response
data = response.json()
print(f"Found {data['total_count']} repositories")

# Check success
if response.status_code == 200:
    print("Success!")
else:
    print(f"Error: {response.status_code}")

❌ Common Mistakes

1. Not checking status code: Always verify response.status_code before using data

2. Calling .json() on non-JSON: Check Content-Type header first or use try/except

3. Hardcoding URLs: Use variables/config for URLs, easier to update and test

✅ Quick Revision

  • requests.get(url) - Send GET request
  • response.status_code - HTTP status (200 = success)
  • response.text - Response body as string
  • response.json() - Parse JSON response
  • response.headers - Response headers dict
  • Use params= for query strings

C.2 POST Requests

HTTP POST for Data Submission

requests.post() sends data to server. Use data= for form data, json= for JSON payloads. POST creates/modifies resources, not idempotent (repeated calls have different effects). Returns Response like GET.

💡 What Is It?

requests.post(url, data=...) or json=... sends HTTP POST request with body data. Used for form submissions, API calls.

🔧 Why It Exists?

GET has limitations: URL length, visible params, no body. POST sends data in body, handles large/sensitive data, modifies state.

⚙️ How It Works?

Sends data in request body. data= for form-encoded, json= for JSON. Server processes, returns response.

🎯 Where Used?

Form submissions, API data creation, file uploads, authentication, webhooks, creating resources.

✅ When To Use?

Use POST for: creating data, submitting forms, sending large/sensitive data, non-idempotent operations.

⚠️ When To Avoid?

Don't use POST for simple data retrieval - use GET. POST not cacheable, can't bookmark.

    'name': 'Alice',

C.3 Headers & Params

🔧 Why It Exists?

APIs need: authentication (headers), filtering (params), content negotiation (headers), pagination (params). Standard HTTP features.

⚙️ How It Works?

params= encoded into URL query string. headers= sent as HTTP headers. Both key-value pairs.

🎯 Where Used?

API authentication, search/filtering, pagination, content negotiation, rate limiting, caching control.

✅ When To Use?

Headers for: auth tokens, API keys, content-type. Params for: search, filters, pagination, sorting.

⚠️ When To Avoid?

Don't put sensitive data in params (visible in URLs/logs). Don't hardcode API keys in headers.

C.4 Status Codes & Response Handling

HTTP Status Codes and Error Management

Status codes indicate request outcome: 2xx (success), 3xx (redirect), 4xx (client error), 5xx (server error). Use response.raise_for_status() to raise exceptions on errors. Check status_code before processing response.

💡 What Is It?

response.status_code is HTTP status: 200 OK, 404 Not Found, 500 Server Error. raise_for_status() throws exception on 4xx/5xx.

🔧 Why It Exists?

Standard way to communicate success/failure. Enables proper error handling, logging, retries, debugging.

⚙️ How It Works?

Server returns status code with response. Client checks code to determine action: process data, retry, show error.

🎯 Where Used?

Error handling, retry logic, monitoring, logging, debugging API issues, conditional processing.

✅ When To Use?

Always check status before using response data. Use raise_for_status() for automatic error handling.

⚠️ When To Avoid?

Don't ignore errors. Don't assume 200 - check explicitly. Some APIs use 200 with error in body.

Simple Example

import requests

# Check status code manually
response = requests.get('https://api.github.com/users/octocat')
if response.status_code == 200:
    data = response.json()
    print("Success:", data['login'])
elif response.status_code == 404:
    print("User not found")
else:
    print(f"Error: {response.status_code}")

# Using raise_for_status()
try:
    response = requests.get('https://api.github.com/invalid')
    response.raise_for_status()  # Raises HTTPError on 4xx/5xx
    data = response.json()
except requests.exceptions.HTTPError as e:
    print(f"HTTP Error: {e}")
except requests.exceptions.RequestException as e:
    print(f"Request failed: {e}")

# Common status codes
# 200 - OK, 201 - Created, 204 - No Content
# 400 - Bad Request, 401 - Unauthorized, 404 - Not Found
# 500 - Server Error, 503 - Service Unavailable

❌ Common Mistakes

1. Not checking status: Always verify status_code before using response data

2. Only checking for 200: 201, 204 are also success. Use response.ok or check >= 200 and < 300

3. Not handling timeouts: Network errors need separate exception handling

✅ Quick Revision

  • 2xx - Success (200 OK, 201 Created, 204 No Content)
  • 3xx - Redirection (301 Moved, 304 Not Modified)
  • 4xx - Client Error (400 Bad Request, 401 Unauthorized, 404 Not Found)
  • 5xx - Server Error (500 Internal Error, 503 Unavailable)
  • response.ok - True if status < 400
  • raise_for_status() - Raises exception on errors

C.5 Timeouts & Exceptions

Network Error Handling and Timeouts

Always set timeout= to prevent hanging forever. Requests raises various exceptions: Timeout, ConnectionError, HTTPError, RequestException (base). Use try/except to handle network failures gracefully.

💡 What Is It?

timeout= (seconds) prevents infinite waiting. Exceptions: Timeout, ConnectionError, HTTPError, RequestException for all errors.

🔧 Why It Exists?

Networks are unreliable. Servers can hang, connections fail, DNS fails. Need timeouts and error handling to prevent app freezing.

⚙️ How It Works?

timeout= waits specified seconds, then raises Timeout. Exceptions provide error details for logging/retry logic.

🎯 Where Used?

Production APIs, web scraping, microservices, external integrations, health checks, anywhere network calls happen.

✅ When To Use?

Always set timeout (default: 5-10 seconds). Catch exceptions for robust error handling. Implement retries for transient failures.

⚠️ When To Avoid?

Don't set timeout too low (causes false failures). Don't silently ignore exceptions. Don't retry indefinitely.

Simple Example

import requests
from requests.exceptions import Timeout, ConnectionError, HTTPError

# Always set timeout
try:
    response = requests.get('https://api.example.com/data', 
                              timeout=5)  # 5 seconds
    response.raise_for_status()
    data = response.json()
    
except Timeout:
    print("Request timed out after 5 seconds")
    
except ConnectionError:
    print("Network connection failed")
    
except HTTPError as e:
    print(f"HTTP error occurred: {e.response.status_code}")
    
except requests.exceptions.RequestException as e:
    print(f"Request failed: {e}")

# Separate connect and read timeouts
response = requests.get('https://api.example.com',
                           timeout=(3, 10))  # (connect, read)

# Retry logic with exponential backoff
import time
for attempt in range(3):
    try:
        response = requests.get('https://api.example.com', timeout=5)
        response.raise_for_status()
        break
    except (Timeout, ConnectionError):
        if attempt < 2:
            time.sleep(2 ** attempt)  # 1s, 2s, 4s
        else:
            print("All retries failed")

❌ Common Mistakes

1. No timeout: Request can hang forever. Always set timeout parameter.

2. Catching generic Exception: Catch specific requests exceptions for better error handling

3. Infinite retries: Limit retry attempts, use exponential backoff to avoid overwhelming servers

✅ Quick Revision

  • timeout=seconds - Prevent hanging (default: None = wait forever)
  • Timeout - Request took too long
  • ConnectionError - Network connection failed
  • HTTPError - 4xx/5xx status codes
  • RequestException - Base exception, catches all
  • Always use timeout= in production
  • Implement retry logic for transient failures

D.2 Finding Elements

Locating Elements on Web Pages

find_element() returns first match, find_elements() returns list. Locators: By.ID, By.CLASS_NAME, By.CSS_SELECTOR, By.XPATH. CSS selectors are faster and more readable than XPath. Raises NoSuchElementException if not found.

💡 What Is It?

Methods to locate elements: find_element(By.ID, 'id'), find_elements(By.CLASS_NAME, 'class'). Returns WebElement or list.

🔧 Why It Exists?

Need to interact with specific page elements. Multiple locator strategies handle different HTML structures and requirements.

⚙️ How It Works?

Searches DOM using specified strategy. find_element returns first match, find_elements returns all matches as list.

🎯 Where Used?

Form automation, testing, scraping, clicking buttons, reading text, extracting data from tables/lists.

✅ When To Use?

CSS_SELECTOR for most cases (fast, readable). XPATH for complex traversal. ID when available (fastest).

⚠️ When To Avoid?

Don't use CLASS_NAME with spaces. Avoid fragile XPath (use relative, not absolute). Don't search before page loads.

Simple Example

from selenium import webdriver
from selenium.webdriver.common.by import By

driver = webdriver.Chrome()
driver.get('https://www.example.com')

element = driver.find_element(By.ID, 'username')

# By CSS Selector (recommended)
element = driver.find_element(By.CSS_SELECTOR, '.button-primary')

element = driver.find_element(By.XPATH, '//button[@type="submit"]')

# By Name attribute
element = driver.find_element(By.NAME, 'email')

# Extract data
text = element.text
attribute = element.get_attribute('href')

driver.quit()

❌ Common Mistakes

1. NoSuchElementException: Element not found. Use explicit waits or check if element exists

2. CLASS_NAME with spaces: Use CSS_SELECTOR instead: ".class1.class2"

3. Absolute XPath: Fragile, breaks with HTML changes. Use relative XPath

✅ Quick Revision

  • By.ID - Fastest, most reliable (if ID exists)
  • By.CSS_SELECTOR - Recommended for most cases
  • By.CLASS_NAME - Single class only (no spaces)
  • By.XPATH - Complex queries, text matching
  • By.NAME - Form inputs with name attribute
  • find_element returns first, find_elements returns list

D.4 Waits (implicit, explicit)

Synchronization Strategies

Implicit waits poll DOM for specified time. Explicit waits use conditions (element_to_be_clickable, presence_of_element). WebDriverWait with expected_conditions is recommended. Avoid time.sleep() - not reliable for dynamic content.

💡 What Is It?

Implicit: driver.implicitly_wait(10) waits globally. Explicit: WebDriverWait(driver, 10).until(condition) waits for specific condition.

🔧 Why It Exists?

Pages load asynchronously, JavaScript renders content dynamically. Need to wait for elements before interacting.

⚙️ How It Works?

Implicit polls DOM every 0.5s up to timeout. Explicit checks condition repeatedly until true or timeout.

🎯 Where Used?

AJAX content, SPAs, loading animations, dynamic forms, lazy-loaded images, infinite scroll.

✅ When To Use?

Use explicit waits with specific conditions. More reliable than implicit. Set implicit wait once at start as fallback.

⚠️ When To Avoid?

Don't mix implicit and explicit waits (conflicts). Don't use time.sleep() for synchronization.

Simple Example

from selenium import webdriver
from selenium.webdriver.common.by import By
from selenium.webdriver.support.ui import WebDriverWait
from selenium.webdriver.support import expected_conditions as EC

driver = webdriver.Chrome()

# Implicit wait (set once, applies globally)
driver.implicitly_wait(10)  # 10 seconds

driver.get('https://www.example.com')

# Explicit wait - element to be clickable
wait = WebDriverWait(driver, 10)
button = wait.until(
    EC.element_to_be_clickable((By.ID, 'submit-btn'))
)
button.click()

# Wait for element presence
element = wait.until(
    EC.presence_of_element_located((By.CLASS_NAME, 'result'))
)

# Wait for visibility
element = wait.until(
    EC.visibility_of_element_located((By.CSS_SELECTOR, 'div.modal'))
)

# Wait for text in element
wait.until(
    EC.text_to_be_present_in_element((By.ID, 'status'), 'Complete')
)

driver.quit()

❌ Common Mistakes

1. Using time.sleep(): Fixed delays are unreliable. Use explicit waits instead

2. Mixing wait types: Don't use implicit and explicit waits together - causes confusion

3. Wrong condition: presence_of_element doesn't mean clickable. Use element_to_be_clickable

✅ Quick Revision

  • Implicit wait - Global timeout for all find operations
  • Explicit wait - Condition-based, more reliable
  • element_to_be_clickable - Element visible and enabled
  • presence_of_element_located - Element in DOM
  • visibility_of_element_located - Element visible on page
  • Prefer explicit waits over implicit

1. *args & **kwargs

Variable-Length Arguments

*args captures positional arguments as tuple, **kwargs captures keyword arguments as dict. Enables flexible function signatures accepting any number of arguments. *args must come before **kwargs. Use when argument count is unknown or for wrapper/decorator functions.

💡 What Is It?

*args collects extra positional args into tuple. **kwargs collects keyword args into dict. Asterisk unpacks sequences/dicts.

🔧 Why It Exists?

Functions need flexibility for unknown arg counts. Enables forwarding args to other functions, creating wrappers, building APIs.

⚙️ How It Works?

Single * unpacks iterables, double ** unpacks dicts. In function def, collects extra args. In calls, unpacks collections.

🎯 Where Used?

Decorators, wrappers, APIs with optional params, logging functions, factory patterns, inheritance with super().

✅ When To Use?

Use when arg count varies, forwarding args, decorators, extending functions. Keep explicit params for required args.

⚠️ When To Avoid?

Don't overuse - makes function signatures unclear. Use explicit params when possible. Document what *args/**kwargs expect.

Simple Example

# *args - positional arguments
def sum_all(*args):
    return sum(args)

print(sum_all(1, 2, 3))        # 6
print(sum_all(1, 2, 3, 4, 5))  # 15

# **kwargs - keyword arguments
def user_info(**kwargs):
    for key, value in kwargs.items():
        print(f"{key}: {value}")

user_info(name="Alice", age=30, city="NYC")

# Combined - order matters!
def combined(required, *args, default=10, **kwargs):
    print(f"Required: {required}")
    print(f"Args: {args}")
    print(f"Default: {default}")
    print(f"Kwargs: {kwargs}")

combined(1, 2, 3, default=20, extra="value")

# Unpacking in function calls
numbers = [1, 2, 3]
print(sum_all(*numbers))  # Unpacks list

data = {'name': 'Bob', 'age': 25}
user_info(**data)  # Unpacks dict
6 15 name: Alice age: 30 city: NYC Required: 1 Args: (2, 3) Default: 20 Kwargs: {'extra': 'value'} 6 name: Bob age: 25

❌ Common Mistakes

1. Wrong order: **kwargs must come last. Order: required, *args, keyword-only, **kwargs

2. Forgetting asterisks: args is just a name, *args is special. Need * to collect/unpack

3. Modifying args tuple: args is tuple (immutable). Convert to list if needed

✅ Quick Revision

  • *args - Collects extra positional args as tuple
  • **kwargs - Collects keyword args as dict
  • Order: required, *args, keyword-only, **kwargs
  • Use * to unpack in calls: func(*list, **dict)
  • Common names: *args, **kwargs (convention, not required)

2. Lambda Functions

Anonymous Single-Expression Functions

Lambda creates small anonymous functions in one line. Syntax: lambda args: expression. Returns expression result automatically. Used with map/filter/sorted for simple operations. Can't contain statements or annotations - only expressions.

💡 What Is It?

lambda x: x*2 creates anonymous function. Single expression, auto-return. Inline function without def/name.

🔧 Why It Exists?

Short callbacks don't need full function definition. Saves lines for simple operations in functional programming.

⚙️ How It Works?

Creates function object without binding to name. Expression evaluated and returned. No statements allowed.

🎯 Where Used?

sorted() key parameter, map/filter, event handlers, simple callbacks, data transformations.

✅ When To Use?

Use for simple one-liners with map/filter/sorted. When function is used once. Keeps code concise.

⚠️ When To Avoid?

Don't use for complex logic. Not for multiple statements. Hurts readability if nested. Use def for reusable functions.

Simple Example

print(add(3, 4))  # 7

# With sorted() - sort by second element
pairs = [(1, 5), (3, 2), (2, 8)]

3. map(), filter(), zip()

Functional Programming Built-ins

map() applies function to every item. filter() keeps items matching condition. zip() combines multiple iterables. All return iterators (lazy evaluation). Convert to list if needed. More memory-efficient than list comprehensions for large data.

💡 What Is It?

map(func, iterable) applies func to all. filter(func, iterable) keeps True items. zip(*iterables) pairs elements.

🔧 Why It Exists?

Functional programming patterns. Cleaner than loops for transformations/filtering. Composable and lazy.

⚙️ How It Works?

Return iterators (lazy). Evaluated on-demand. map/filter exhaust after one use. zip stops at shortest iterable.

🎯 Where Used?

Data transformation pipelines, filtering, parallel iteration, ETL processes, functional-style code.

✅ When To Use?

Use map for transformations, filter for conditions, zip for parallel iteration. Good with lambda/existing functions.

⚠️ When To Avoid?

List comprehensions often more readable. Don't use if you need intermediate values. Not for complex logic.

Simple Example

4. List Comprehensions

Concise List Construction

[expression for item in iterable if condition] creates lists in one line. More readable than map/filter for many cases. Can nest and include conditionals. Eager evaluation (computes immediately). Faster than equivalent loops.

💡 What Is It?

[expr for x in iter if cond] - Creates list with expression evaluated for filtered items. Pythonic syntax.

🔧 Why It Exists?

Cleaner than for loops for list construction. More readable than map/filter. Combines transformation + filtering.

⚙️ How It Works?

Iterates collection, applies optional filter, evaluates expression, collects results. Eager (creates list immediately).

🎯 Where Used?

Data transformation, filtering, flattening nested lists, parsing, ETL, creating derived collections.

✅ When To Use?

Use for simple transformations/filtering. When result is small/medium list. More readable than loops.

⚠️ When To Avoid?

Don't use for complex logic (use loops). Not for large data (use generators). Avoid deep nesting.

Simple Example

1. If-else syntax: [x if cond else y for...] (if-else before for), [x for... if cond] (filter only)

5. Dict & Set Comprehensions

Dictionary and Set Construction Syntax

Dict comprehension: {key: value for x in iter}. Set comprehension: {expr for x in iter}. Similar to list comprehensions but create dicts/sets. Sets automatically deduplicate. Dicts handle key collisions (last wins).

💡 What Is It?

{k:v for x in iter} creates dict. {expr for x in iter} creates set. Curly braces with comprehension syntax.

🔧 Why It Exists?

Concise dict/set creation. Transform data into key-value mappings. Automatic deduplication for sets.

⚙️ How It Works?

Dict: evaluates key and value expressions. Set: evaluates expression, deduplicates. Both support filtering.

🎯 Where Used?

Creating lookups, inverting dicts, grouping data, deduplication, building indexes, data transformation.

✅ When To Use?

Dict: key-value transformations, lookups. Set: removing duplicates, membership testing, unique values.

⚠️ When To Avoid?

Don't use for complex logic. Watch for dict key collisions. Set values must be hashable.

Simple Example

# Set comprehension - remove duplicates
numbers = [1, 2, 2, 3, 3, 3, 4]
unique = {x for x in numbers}

1. Dict key collisions: Duplicate keys - last value wins. Check for duplicates first

6. Generator Expressions

(expr for x in iter) creates generator - like list comprehension but with parentheses. Lazy evaluation: computes on-demand. Memory efficient for large datasets. One-time iteration. Use when you don't need full list in memory.

7. Closures

Functions with Enclosed Scope

Closure: inner function accessing outer function's variables after outer returns. Captures and remembers enclosing scope. Creates private state without classes. Used for function factories, decorators, callbacks with state.

💡 What Is It?

Inner function that remembers variables from outer function's scope, even after outer function returns.

🔧 Why It Exists?

Creates functions with private state. Alternative to classes for simple state management. Enables function factories.

⚙️ How It Works?

Inner function stores reference to outer scope variables. Variables persist in memory via __closure__ attribute.

🎯 Where Used?

Decorators, function factories, callbacks with state, data hiding, configuration functions.

✅ When To Use?

Use for simple state encapsulation, function customization, avoiding global state, lightweight alternatives to classes.

⚠️ When To Avoid?

Don't use for complex state management - use classes. Watch for memory leaks if closure holds large objects.

Simple Example

# Basic closure - counter
def make_counter():
    count = 0  # Enclosed variable
    
    def counter():
        nonlocal count  # Modify outer variable
        count += 1
        return count
    
    return counter

c1 = make_counter()
print(c1())  # 1
print(c1())  # 2
print(c1())  # 3

c2 = make_counter()  # Independent counter
print(c2())  # 1

def make_multiplier(factor):
    def multiply(x):
        return x * factor
    return multiply

print(times5(10))  # 50

print(hola("Bob"))     # Hola, Bob!

❌ Common Mistakes

1. Forgetting nonlocal: Need nonlocal to modify outer variables, not just read

2. Loop variable capture: Closures in loops capture reference, not value. Use default args

3. Memory leaks: Closures keep outer scope alive. Clear references when done

✅ Quick Revision

  • Closure = inner function + enclosed scope
  • nonlocal keyword to modify outer variables
  • Creates private state without classes
  • Each closure instance has independent state
  • Used in decorators and function factories
  • Variables stored in __closure__ attribute

8. Iterators

Objects Supporting Iteration Protocol

Iterator: object with __iter__() and __next__() methods. __iter__() returns self, __next__() returns next value or raises StopIteration. Iterable has __iter__() returning iterator. All iterators are iterable. Used by for loops, comprehensions.

💡 What Is It?

Iterator: object with __next__() method. Returns one value at a time, raises StopIteration when done.

🔧 Why It Exists?

Uniform interface for sequential access. Memory efficient - one item at a time. Enables lazy evaluation.

⚙️ How It Works?

iter(obj) calls __iter__(), returns iterator. next(iter) calls __next__(). StopIteration signals end.

🎯 Where Used?

For loops, comprehensions, unpacking, built-in functions (sum, max), custom iteration logic.

✅ When To Use?

Create custom iterators for: complex sequences, lazy computation, infinite streams, stateful iteration.

⚠️ When To Avoid?

Don't implement manually if generator works. Generators are simpler. Use iterators for complex state.

class Countdown:
    def __init__(self, start):
        self.current = start
    
    def __iter__(self):
        return self
    
    def __next__(self):
        if self.current <= 0:
            raise StopIteration
        self.current -= 1
        return self.current + 1

for num in cd:
    print(num, end=' ')

        return self

3. One-time use: Iterators exhaust. Create new instance or make iterable separate from iterator

  • __iter__() - Returns iterator (usually self)
  • __next__() - Returns next value or raises StopIteration

  • Iterators are stateful and one-time use

9. Generators & yield

Functions That Pause and Resume

Generator function uses yield instead of return. Pauses execution, saves state, resumes on next call. Returns generator iterator automatically. Simpler than manual iterator classes. Memory efficient for sequences.

💡 What Is It?

Function with yield keyword. Creates generator iterator. Pauses at yield, resumes on next(). State preserved between calls.

🔧 Why It Exists?

Simpler than iterator classes. Automatic state management. Cleaner code for lazy sequences. Memory efficient.

⚙️ How It Works?

yield returns value and pauses. Next call resumes after yield. Local variables preserved. StopIteration on return/end.

🎯 Where Used?

File processing, infinite sequences, pipelines, lazy evaluation, streaming data, memory-efficient iteration.

✅ When To Use?

Use for sequences, lazy computation, large data, state between iterations. Cleaner than iterator classes.

⚠️ When To Avoid?

Don't use if need random access or length. Not for reusable sequences (exhaust after one use).

Simple Example

    while n > 0:
        yield n

10. yield from

🔧 Why It Exists?

Cleaner syntax for delegation. Handles send/throw correctly. Enables recursive generators. Flattens nested structures.

⚙️ How It Works?

Yields all values from sub-iterable. Forwards generator methods (send, throw, close) to sub-generator.

🎯 Where Used?

Flattening nested lists, tree traversal, recursive generators, composing generators, delegation patterns.

✅ When To Use?

Use for generator composition, flattening, recursive structures. Cleaner than manual loops.

⚠️ When To Avoid?

Don't use if need to transform values while yielding. Use regular yield in loop instead.

11. Function Decorators

Functions That Modify Other Functions

Decorator: function that takes function and returns modified version. Syntax: @decorator above function definition. Wraps original function with additional behavior. Common uses: logging, timing, authentication, caching.

💡 What Is It?

@decorator syntax wraps function. Decorator takes func, returns wrapper that adds behavior before/after original call.

🔧 Why It Exists?

Adds functionality without modifying original code. DRY principle - reusable behavior. Separation of concerns.

⚙️ How It Works?

@decorator is syntactic sugar for: func = decorator(func). Wrapper function captures original, adds behavior.

🎯 Where Used?

Logging, timing, authentication, caching, validation, retry logic, web frameworks (Flask routes).

✅ When To Use?

Use for cross-cutting concerns: logging, timing, access control. When same behavior needed across many functions.

⚠️ When To Avoid?

Don't overuse - can make code hard to debug. Don't use for function-specific logic. Keep decorators simple.

Simple Example

# Basic decorator - logging
def log_calls(func):
    def wrapper(*args, **kwargs):
        print(f"Calling {func.__name__} with {args}, {kwargs}")
        result = func(*args, **kwargs)
        print(f"{func.__name__} returned {result}")
        return result
    return wrapper

@log_calls
def add(a, b):
    return a + b

def timer(func):
    def wrapper(*args, **kwargs):
        start = time.time()
        result = func(*args, **kwargs)
        end = time.time()
        print(f"{func.__name__} took {end - start:.4f}s")
        return result
    return wrapper

@timer
def slow_function():
    time.sleep(1)
    return "Done"

@log_calls

    return x * y

12. Decorators with Args

Decorator with arguments requires three levels: outer function takes decorator args, middle function takes target function, inner wrapper executes target. @decorator(args) calls outer, returns middle, which decorates function.

💡 What Is It?

🔧 Why It Exists?

Configurable decorators. Same behavior with different parameters. E.g., @repeat(3) vs @repeat(5).

⚙️ How It Works?

Outer function captures decorator args, returns actual decorator. Middle returns wrapper. Three closures nested.

🎯 Where Used?

Retry with attempts, rate limiting, timeout values, permission levels, repeat counts.

✅ When To Use?

Use when decorator needs configuration. Makes decorators reusable with different settings.

⚠️ When To Avoid?

Don't use if no parameters needed - adds unnecessary complexity. Keep simple decorators simple.

        def wrapper(*args, **kwargs):

Every decorator that wraps functions. Documentation tools, debuggers, IDEs rely on proper metadata.

15. functools.lru_cache

Memoization with LRU Eviction

@lru_cache(maxsize=128) caches function results. LRU = Least Recently Used eviction when cache full. Dramatic speedup for expensive, pure functions with repeated arguments. Thread-safe. Arguments must be hashable.

💡 What Is It?

@lru_cache memoizes function results. Stores (args → result) mapping. Returns cached value on repeat calls.

🔧 Why It Exists?

Avoid recomputing expensive operations. Optimization for pure functions. Trades memory for speed.

⚙️ How It Works?

Hash arguments as key, store result. On call, check cache first. LRU evicts oldest when maxsize reached.

🎯 Where Used?

Recursive functions (Fibonacci), expensive calculations, API calls, database queries, repeated computations.

✅ When To Use?

Use for pure functions with expensive computation and repeated calls. Perfect for recursion.

⚠️ When To Avoid?

Don't use with side effects, I/O, random results. Arguments must be hashable. Watch memory usage.

Simple Example

from functools import lru_cache

    if n < 2:
        return n
    return fib_slow(n-1) + fib_slow(n-2)

print(fib_slow(30))

# Fibonacci with cache - fast!
@lru_cache(maxsize=128)

    if n < 2:
        return n
    return fib_fast(n-1) + fib_fast(n-2)

start = time.time()
print(fib_fast(30))
print(f"Fast: {time.time() - start:.6f}s")

    time.sleep(1)  # Simulate expensive operation
    return x ** y

        return 1

16. Custom Context Managers

Resource Management with __enter__ and __exit__

Context managers handle setup/teardown automatically. Implement __enter__() and __exit__() methods, or use @contextmanager decorator. __enter__ runs on 'with' entry, __exit__ runs on exit (even if exception). Guarantees cleanup.

💡 What Is It?

Class with __enter__ and __exit__, or function with @contextmanager. Used with 'with' statement for automatic resource management.

🔧 Why It Exists?

Ensures cleanup happens: files close, locks release, connections close. Even if exceptions occur. Prevents resource leaks.

⚙️ How It Works?

__enter__ executes on 'with' entry, returns resource. __exit__ executes on exit, receives exception info if any.

🎯 Where Used?

File operations, database connections, locks, timers, temporary changes, transaction management.

✅ When To Use?

Use when need guaranteed cleanup: files, connections, locks, temporary state. Makes code safer and cleaner.

⚠️ When To Avoid?

Don't use for simple operations without cleanup. Don't suppress exceptions unnecessarily in __exit__.

Simple Example

# Class-based context manager
class FileManager:
    def __init__(self, filename, mode):
        self.filename = filename
        self.mode = mode
        self.file = None
    
    def __enter__(self):
        print("Opening file")
        self.file = open(self.filename, self.mode)
        return self.file
    
    def __exit__(self, exc_type, exc_val, exc_tb):
        print("Closing file")
        if self.file:
            self.file.close()
        return False  # Don't suppress exceptions

with FileManager('test.txt', 'w') as f:
    f.write('Hello World')

19. Circular Imports

Import Dependency Cycles

Circular import: A imports B, B imports A. Causes ImportError or AttributeError. Solutions: move imports to functions, refactor to third module, use TYPE_CHECKING for type hints, restructure dependencies.

💡 What Is It?

Module A imports B, B imports A. Creates dependency cycle. Python can't fully import both modules.

🔧 Why It Exists?

Poor architecture - tight coupling between modules. Often indicates need for refactoring.

⚙️ How It Works?

Python partially imports A, then tries B. B imports A (partially loaded), causes AttributeError.

🎯 Where Used?

Common in poorly structured projects, tight coupling, mutual dependencies, shared utilities.

✅ When To Use?

Never intentionally. Fix via: local imports, refactoring, dependency injection, interface modules.

⚠️ When To Avoid?

Always avoid. Sign of design flaw. Refactor to eliminate cycles.

Simple Example

# === PROBLEM: Circular import ===

# module_a.py
from module_b import func_b

def func_a():
    return "A" + func_b()

# module_b.py
from module_a import func_a  # Circular!

def func_b():
    return "B" + func_a()

# Result: ImportError or AttributeError

# === SOLUTION 1: Local imports ===

# module_a.py
def func_a():
    from module_b import func_b  # Import inside function
    return "A" + func_b()

# module_b.py
def func_b():
    from module_a import func_a
    return "B"

# === SOLUTION 2: Refactor to third module ===

# shared.py
def shared_func():
    return "Shared"

# module_a.py
from shared import shared_func

def func_a():
    return "A" + shared_func()

# module_b.py
from shared import shared_func

def func_b():
    return "B" + shared_func()

# === SOLUTION 3: TYPE_CHECKING (for type hints) ===

# module_a.py
from typing import TYPE_CHECKING

if TYPE_CHECKING:
    from module_b import ClassB  # Only for type checker

def process(obj: 'ClassB') -> str:  # String annotation
    return obj.name

# === SOLUTION 4: Dependency injection ===

# module_a.py
def func_a(func_b_impl):  # Pass dependency
    return "A" + func_b_impl()

# module_b.py
def func_b():
    return "B"

# main.py
from module_a import func_a
from module_b import func_b

result = func_a(func_b)  # Inject dependency

❌ Common Mistakes

1. Ignoring the problem: Circular imports indicate design issues. Fix the architecture

2. Overusing local imports: Quick fix but hurts performance. Better to refactor

3. Complex dependency graphs: Multiple circular dependencies. Need major refactoring

✅ Quick Revision

  • Circular import: A imports B, B imports A
  • Solution 1: Local imports (in functions)
  • Solution 2: Extract to third module
  • Solution 3: TYPE_CHECKING for type hints
  • Solution 4: Dependency injection
  • Best fix: Refactor architecture to eliminate cycle

22. Method Resolution Order

Inheritance Search Path (C3 Linearization)

MRO determines method lookup order in inheritance. Uses C3 linearization algorithm. View with Class.__mro__ or Class.mro(). Left-to-right, depth-first, but preserves local precedence. Critical for multiple inheritance.

💡 What Is It?

Order Python searches for methods in inheritance hierarchy. Tuple of classes: Class.__mro__.

🔧 Why It Exists?

Multiple inheritance needs consistent method resolution. Prevents ambiguity. Diamond problem solution.

⚙️ How It Works?

C3 linearization algorithm. Respects inheritance order, preserves monotonicity, follows DFS with left-to-right.

🎯 Where Used?

Multiple inheritance, mixins, understanding super(), debugging method calls, framework design.

✅ When To Use?

Check MRO when using multiple inheritance. Use super() for cooperative inheritance. Design mixins carefully.

⚠️ When To Avoid?

Don't create complex hierarchies. Keep MRO simple. Avoid diamond inheritance when possible.

Simple Example

# Simple inheritance
class A:
    def method(self):
        print("A")

class B(A):
    def method(self):
        print("B")

print(B.__mro__)  # (, , )

# Multiple inheritance - diamond problem
class Base:
    def method(self):
        print("Base")

class Left(Base):
    def method(self):
        print("Left")

class Right(Base):
    def method(self):
        print("Right")

class Child(Left, Right):  # Left first
    pass

print(Child.__mro__)
# (, , , , )

c = Child()
c.method()  # Left - uses MRO

# Using super() with MRO
class A:
    def method(self):
        print("A")

class B(A):
    def method(self):
        print("B")
        super().method()  # Calls next in MRO

class C(A):
    def method(self):
        print("C")
        super().method()

class D(B, C):
    def method(self):
        print("D")
        super().method()

print(D.__mro__)
# (, , , , )

d = D()
d.method()

(, , ) (, , , , ) Left (, , , , ) D B C A

❌ Common Mistakes

1. Ignoring MRO: Unexpected method calls in multiple inheritance. Always check MRO

2. Wrong super() usage: Don't use ClassName.method(self). Use super() for MRO cooperation

3. Complex hierarchies: Deep multiple inheritance hard to maintain. Keep simple

✅ Quick Revision

  • Class.__mro__ - View method resolution order
  • C3 linearization algorithm
  • Left-to-right, depth-first search
  • super() follows MRO, not just parent
  • Diamond problem solved by MRO
  • All classes inherit from object

34. asyncio / async / await

Asynchronous I/O with Coroutines

asyncio: single-threaded concurrency. async def creates coroutine. await suspends execution. Event loop schedules coroutines. Perfect for I/O-bound tasks. Non-blocking operations.

💡 What Is It?

async/await syntax for coroutines. asyncio event loop schedules concurrent tasks. Single-threaded async I/O.

🔧 Why It Exists?

Threading has overhead. Async single-threaded but concurrent. Better for many I/O operations. Lower resource usage.

⚙️ How It Works?

Event loop runs coroutines. await yields control. Loop switches between tasks during I/O waits.

🎯 Where Used?

Web servers, API clients, websockets, database queries, concurrent network I/O, modern web frameworks.

✅ When To Use?

Use for I/O-bound with many concurrent operations. Web scraping, API calls, network services.

⚠️ When To Avoid?

Don't use for CPU-bound tasks. Blocking operations block everything. Requires async libraries.

Simple Example

import asyncio

# Basic coroutine
async def say_hello():
    print("Hello")
    await asyncio.sleep(1)  # Non-blocking sleep
    print("World")

# Run coroutine
asyncio.run(say_hello())

# Multiple concurrent tasks
async def fetch_data(id):
    print(f"Fetching {id}...")
    await asyncio.sleep(2)  # Simulate I/O
    print(f"Done {id}")

async def main():
    # Run concurrently
    results = await asyncio.gather(
        fetch_data(1),
        fetch_data(2),
        fetch_data(3)
    )
    print(results)

    result1 = await task1
    result2 = await task2

35. Event Loops & Tasks

Understanding asyncio Event Loop

Event loop is asyncio core. Schedules and executes coroutines. asyncio.get_event_loop(), loop.run_until_complete(). Tasks wrap coroutines for scheduling. asyncio.create_task() schedules execution. Loop manages all async operations.

💡 What Is It?

Event loop: heart of asyncio. Schedules coroutines, manages I/O, switches between tasks during waits.

🔧 Why It Exists?

Single-threaded concurrency needs scheduler. Event loop coordinates multiple coroutines efficiently.

⚙️ How It Works?

Loop maintains ready queue. Executes tasks until await. Switches to next ready task. Efficient context switching.

🎯 Where Used?

All asyncio programs. Web frameworks (FastAPI, aiohttp), async libraries, concurrent I/O operations.

✅ When To Use?

Understanding event loop helps debug. Custom loop policies for special cases. Task management.

⚠️ When To Avoid?

Don't manually manage loop usually. asyncio.run() handles it. Avoid blocking the loop.

Simple Example

    r1 = await task1
    r2 = await task2
    print(r1, r2)

36. Futures & Executors

💡 What Is It?

Executor: high-level interface for async execution. Future: pending result. Unified API for threads/processes.

🔧 Why It Exists?

Simpler than raw threading/multiprocessing. Consistent interface. Better resource management with pools.

⚙️ How It Works?

Executor manages worker pool. submit() schedules task, returns Future. result() waits for completion.

🎯 Where Used?

Parallel tasks, concurrent I/O, batch processing, background jobs, worker pools.

✅ When To Use?

Need thread/process pools. Want simple concurrent API. Managing multiple parallel tasks.

⚠️ When To Avoid?

Simple sequential tasks. When asyncio more appropriate. Overhead not worth it.

Simple Example

from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor, as_completed
import time

# ThreadPoolExecutor
def task(n):
    time.sleep(1)
    return n * n

with ThreadPoolExecutor(max_workers=3) as executor:
    future = executor.submit(task, 5)
    print(f"Done: {future.done()}")
    result = future.result()  # Blocks until ready
    print(result)

# Map multiple tasks
with ThreadPoolExecutor() as executor:
    results = executor.map(task, range(5))
    print(list(results))

# Process as they complete
with ThreadPoolExecutor() as executor:
    futures = [executor.submit(task, i) for i in range(5)]
    for future in as_completed(futures):
        print(future.result())

1. Not using context manager: Use with statement for automatic cleanup

2. Calling result() too early: Blocks. Use as_completed() or done()

3. Wrong executor type: Thread for I/O, Process for CPU

  • ThreadPoolExecutor - Thread pool for I/O
  • ProcessPoolExecutor - Process pool for CPU

37. Global Interpreter Lock

Python's Threading Limitation

GIL: mutex preventing multiple threads executing Python bytecode simultaneously. One thread executes at a time. Protects memory management. Makes threading ineffective for CPU-bound. I/O operations release GIL.

💡 What Is It?

Lock ensuring only one thread executes Python code at a time. CPython implementation detail.

🔧 Why It Exists?

Memory management not thread-safe. Reference counting needs protection. Simplifies C API.

⚙️ How It Works?

Thread acquires GIL before executing. Releases after time slice or I/O. Other threads wait.

🎯 Where Used?

CPython interpreter (not Jython, IronPython). Affects all threaded programs.

✅ When To Use?

Understanding helps choose concurrency model. Threading fine for I/O. Use multiprocessing for CPU.

⚠️ When To Avoid?

Can't avoid GIL in CPython. Use multiprocessing or asyncio for CPU/I/O tasks respectively.

Simple Example

import threading
import time

# CPU-bound task (GIL limits performance)
def cpu_task():
    total = 0
    for i in range(10_000_000):
        total += i
    return total

# Sequential
start = time.time()
cpu_task()
cpu_task()
print(f"Sequential: {time.time()-start:.2f}s")

# Threaded (NOT faster due to GIL)
start = time.time()
t1 = threading.Thread(target=cpu_task)
t2 = threading.Thread(target=cpu_task)
t1.start()
t2.start()
t1.join()
t2.join()
print(f"Threaded: {time.time()-start:.2f}s")

# I/O-bound works (GIL released during I/O)
def io_task():
    time.sleep(1)  # Releases GIL

start = time.time()
threads = [threading.Thread(target=io_task) for _ in range(5)]
for t in threads:
    t.start()
for t in threads:
    t.join()
print(f"I/O Threaded: {time.time()-start:.2f}s")  # ~1s, not 5s

❌ Common Mistakes

1. Using threads for CPU: GIL prevents parallelism. Use multiprocessing

2. Assuming parallelism: Threads concurrent, not parallel for CPU tasks

3. Ignoring alternatives: asyncio for I/O, multiprocessing for CPU

✅ Quick Revision

  • GIL allows one thread executing Python at a time
  • Protects memory management (reference counting)
  • Released during I/O operations
  • Threading good for I/O, not CPU
  • Use multiprocessing to bypass GIL
  • Not in Jython, IronPython, PyPy (in some cases)

38. Memory Management

Python's Automatic Memory Management

Reference counting + garbage collection. Each object has refcount. When 0, memory freed. Garbage collector handles circular references. sys.getrefcount(), gc module. Memory pools for efficiency.

💡 What Is It?

Automatic memory allocation/deallocation. Reference counting + GC for cycles. No manual malloc/free.

🔧 Why It Exists?

Prevent memory leaks, simplify development. Automatic cleanup. Handle circular references.

⚙️ How It Works?

Refcount tracks references. GC finds cycles. Memory pools reuse small objects.

🎯 Where Used?

All Python programs. Understanding helps optimize, debug leaks, improve performance.

✅ When To Use?

Monitor with gc module. Use __del__ carefully. Profile memory for large apps.

⚠️ When To Avoid?

Don't rely on __del__ timing. Don't disable GC without reason. Trust automatic management.

Simple Example

import sys
import gc

# Reference counting
x = [1, 2, 3]
print(sys.getrefcount(x))  # 2 (x + getrefcount arg)

y = x  # Reference increases
print(sys.getrefcount(x))  # 3

del y  # Reference decreases
print(sys.getrefcount(x))  # 2

# Garbage collection
print(gc.isenabled())  # True
print(gc.get_count())  # Generation counts

# Circular reference
class Node:
    def __init__(self):
        self.ref = None

a = Node()
b = Node()
a.ref = b  # a -> b
b.ref = a  # b -> a (cycle!)
del a, b  # GC cleans up cycle

1. Circular references: Create memory leaks without GC. Break cycles manually if needed

2. Large data in memory: Use generators, process in chunks

  • GC handles circular references
  • sys.getrefcount() - Check references
  • gc.collect() - Force collection

39. Shallow vs Deep Copy

Copying Objects and Nested Structures

Shallow copy: copy.copy() - copies object, not nested objects. Deep copy: copy.deepcopy() - recursive copy. Assignment creates reference, not copy. Use copy for mutable objects.

💡 What Is It?

Shallow: new container, same nested objects. Deep: recursive copy of all nested objects.

🔧 Why It Exists?

Assignment creates alias. Need independent copies. Nested structures need deep copy.

⚙️ How It Works?

copy() creates new object, copies references. deepcopy() recursively creates new nested objects.

🎯 Where Used?

Copying lists, dicts with nested data. Preventing unintended mutations. Data cloning.

✅ When To Use?

Shallow for flat structures. Deep for nested structures. When need independent copy.

⚠️ When To Avoid?

Deep copy expensive for large structures. Consider sharing immutable data instead.

Simple Example

import copy

# Assignment (reference)
a = [1, [2, 3]]
b = a  # Same object!
b[0] = 99
print(a)  # [99, [2, 3]] - a changed!

# Shallow copy
a = [1, [2, 3]]
b = copy.copy(a)
b[0] = 99
print(a)  # [1, [2, 3]] - a unchanged
b[1][0] = 99
print(a)  # [1, [99, 3]] - nested changed!

# Deep copy
a = [1, [2, 3]]
b = copy.deepcopy(a)
b[1][0] = 99
print(a)  # [1, [2, 3]] - fully independent!

❌ Common Mistakes

1. Using = for copying: Creates reference, not copy

2. Shallow copy nested structures: Nested objects still shared

✅ Quick Revision

  • a = b - Reference (alias)
  • copy.copy() - Shallow copy
  • copy.deepcopy() - Deep recursive copy
  • list.copy(), dict.copy() also shallow

46. Plugin Architectures

Simple Example

from abc import ABC, abstractmethod
import importlib
import pkgutil

# Plugin interface
class Plugin(ABC):
    @abstractmethod
    def execute(self):
        pass

# Plugin implementation
class HelloPlugin(Plugin):
    def execute(self):
        print("Hello from plugin!")

# Plugin manager
class PluginManager:
    def __init__(self):
        self.plugins = []
    
    def register(self, plugin):
        self.plugins.append(plugin)
    
    def execute_all(self):
        for plugin in self.plugins:
            plugin.execute()
    
    def discover_plugins(self, package_name):
        # Discover plugins in package
        for _, name, _ in pkgutil.iter_modules([package_name]):
            module = importlib.import_module(f"{package_name}.{name}")
            for item_name in dir(module):
                item = getattr(module, item_name)
                if isinstance(item, type) and issubclass(item, Plugin) and item is not Plugin:
                    self.register(item())

# Usage
manager = PluginManager()
manager.register(HelloPlugin())
manager.execute_all()

Virtual environment: isolated Python installation. venv module (builtin), virtualenv, conda. Project-specific dependencies. Avoid global package pollution. requirements.txt for reproducibility.

Isolated Python environment. Separate site-packages. Project-specific dependencies.

🔧 Why It Exists?

Creates directory with Python copy. Modifies PATH. Separate site-packages directory.

🎯 Where Used?

Every Python project. Development, testing, production. CI/CD pipelines.

✅ When To Use?

Always! One venv per project. Before installing any packages.

System tools (use pipx). Docker already isolates. Quick scripts (but still recommended).

49. Performance Profiling

Identifying Performance Bottlenecks

Profiling: measure where time spent. cProfile (function-level), line_profiler (line-level), memory_profiler. timeit for microbenchmarks. Profile before optimizing. Visualization with snakeviz, py-spy.

💡 What Is It?

Tools measuring execution time, memory usage. Find bottlenecks. cProfile, timeit, memory_profiler.

🔧 Why It Exists?

Premature optimization wastes time. Profile to find actual bottlenecks. Measure before/after.

⚙️ How It Works?

cProfile: function call counts. line_profiler: line-by-line. memory_profiler: memory usage over time.

🎯 Where Used?

Performance optimization, finding bottlenecks, benchmarking, production profiling.

✅ When To Use?

Before optimization. When performance issues. Benchmark alternatives. Continuous monitoring.

⚠️ When To Avoid?

Don't profile everything. Focus on slow paths. Profiling has overhead.

Simple Example

import cProfile
import pstats
from io import StringIO

# Function to profile
def slow_function():
    total = 0
    for i in range(1000000):
        total += i
    return total

# cProfile - function level
cProfile.run('slow_function()')

# Programmatic profiling
profiler = cProfile.Profile()
profiler.enable()
slow_function()
profiler.disable()

s = StringIO()
ps = pstats.Stats(profiler, stream=s).sort_stats('cumulative')
ps.print_stats(10)  # Top 10
print(s.getvalue())

# timeit - microbenchmarks
import timeit

time = timeit.timeit('sum(range(100))', number=10000)
print(f"Time: {time:.4f}s")

# Compare implementations
list_comp = timeit.timeit('[x**2 for x in range(100)]', number=10000)
map_func = timeit.timeit('list(map(lambda x: x**2, range(100)))', number=10000)
print(f"List comp: {list_comp:.4f}s, Map: {map_func:.4f}s")

# line_profiler (requires installation)
# @profile  # Decorator
# def my_function():
#     ...
# kernprof -l -v script.py

# memory_profiler
# from memory_profiler import profile
# @profile
# def my_function():
#     ...
# python -m memory_profiler script.py
Time: 0.0523s List comp: 0.0821s, Map: 0.1234s

❌ Common Mistakes

1. Optimizing without profiling: Profile first, find actual bottlenecks

2. Microbenchmarks in production: Use realistic data and scenarios

3. Ignoring memory: Profile both time and memory

✅ Quick Revision

  • cProfile - Function-level profiling
  • timeit - Microbenchmarks
  • line_profiler - Line-by-line
  • memory_profiler - Memory usage
  • py-spy - Production profiling (sampling)
  • Always profile before optimizing

50. Testing (unittest, pytest)

Automated Testing Frameworks

unittest: builtin test framework. pytest: modern, feature-rich. Assertions, fixtures, mocking. Test discovery, parameterization. CI/CD integration. Coverage with pytest-cov.

💡 What Is It?

Frameworks for automated testing. unittest (builtin), pytest (modern). Assertions, fixtures, mocking.

🔧 Why It Exists?

Catch bugs early. Regression prevention. Confidence in refactoring. Documentation through tests.

⚙️ How It Works?

Test discovery (test_*.py). Assertions verify behavior. Fixtures setup/teardown. Mocks isolate units.

🎯 Where Used?

All production code. TDD, CI/CD pipelines, regression testing, integration tests.

✅ When To Use?

Always! Write tests alongside code. Before fixing bugs. Critical business logic first.

⚠️ When To Avoid?

Prototypes, one-off scripts. But even then, tests help catch issues early.

Simple Example

# unittest (builtin)
import unittest

def add(a, b):
    return a + b

class TestMath(unittest.TestCase):
    def test_add(self):
        self.assertEqual(add(2, 3), 5)
        self.assertEqual(add(-1, 1), 0)
    
    def test_add_negative(self):
        self.assertEqual(add(-2, -3), -5)

if __name__ == '__main__':
    unittest.main()

# pytest (modern, cleaner)
# test_math.py
def add(a, b):
    return a + b

def test_add():
    assert add(2, 3) == 5
    assert add(-1, 1) == 0

def test_add_negative():
    assert add(-2, -3) == -5

# Pytest fixtures
import pytest

@pytest.fixture
def sample_data():
    return [1, 2, 3, 4, 5]

def test_sum(sample_data):
    assert sum(sample_data) == 15

# Parametrized tests
@pytest.mark.parametrize("a,b,expected", [
    (2, 3, 5),
    (-1, 1, 0),
    (0, 0, 0),
])
def test_add_parametrized(a, b, expected):
    assert add(a, b) == expected

# Mocking
from unittest.mock import Mock, patch

def get_user_data(api):
    return api.fetch_user()

def test_get_user_data():
    mock_api = Mock()
    mock_api.fetch_user.return_value = {"name": "Alice"}
    
    result = get_user_data(mock_api)
    assert result["name"] == "Alice"
    mock_api.fetch_user.assert_called_once()

# Run tests
# python -m pytest  # Discover and run all
# python -m pytest test_math.py  # Specific file
# python -m pytest -v  # Verbose
# python -m pytest --cov=mymodule  # With coverage

❌ Common Mistakes

1. Not testing edge cases: Test empty, null, boundary values

2. Testing implementation: Test behavior, not internals

3. No CI integration: Run tests automatically on every commit

✅ Quick Revision

  • unittest - Builtin, class-based
  • pytest - Modern, function-based, simpler
  • Fixtures - Setup/teardown data
  • Parametrize - Run same test with different inputs
  • Mock - Isolate units, fake dependencies
  • Coverage - Measure test completeness
