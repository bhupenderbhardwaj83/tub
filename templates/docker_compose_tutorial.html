What is Docker Compose?

Docker Compose is a tool for defining and running multi-container Docker applications. With Compose, you use a YAML file to configure your application's services, networks, and volumes.

The Problem

Running `docker run` 10 times for a frontend, backend, database, cache, and proxy is tedious and error-prone. Connecting them manually via networks is even harder.

The Solution

Define the entire stack in one file (`docker-compose.yml`). Spin it all up with `docker compose up`. Tear it down with `docker compose down`.

Core Concepts

📄

YAML File

The blueprint of your application stack.

⚙️

Services

A container in the context of your app (e.g., 'web', 'db').

🔌

Networks

How services communicate. Usually created automatically.

💾

Volumes

Persistent data storage shared or kept across restarts.

3

Architecture

How it works

  • 1
    Project Name

    Compose isolates environments using a project name (usually the directory name). This allows multiple instances of the same code to run on one host.

  • 2
    Dependency Graph

    Compose calculates which services depend on others (using `depends_on`) to start them in the correct order.

  • 3
    Docker API

    The Compose CLI converts YAML config into API calls to the Docker Daemon, creating specific networks, volumes, and containers.

Installation

Compose v2 (Recommended)

Rewritten in Go. It is now a Docker CLI plugin. Syntax is `docker compose (without dash)`.

$ docker compose version
Docker Compose version v2.10.2

Compose v1 (Legacy)

The old Python script. Syntax was `docker-compose (with dash)`. Deprecated.

$ docker-compose version
docker-compose version 1.29.2
5

The Compose File

# 1. Version (Optional in v2)
version: '3.8'

# 2. Services Definition
services:
  web:                            # Service Name
    image: nginx:latest           # Image to pull
    restart: always               # Restart Policy
    ports:
      - "80:80"                   # Host:Container Mapping
    depends_on:                   # Dependency
      - api

  api:
    build: ./api                  # Build from Dockerfile
    environment:                  # Env Vars
      - DEBUG=true
    volumes:                      # Volumes
      - ./api:/app

# 3. Named Volumes
volumes:
  db_data:

# 4. Custom Networks
networks:
  frontend:
build:

Builds an image from a local Dockerfile instead of pulling.

depends_on:

Control startup order. Note: Does not wait for service to be "ready", only "started".

profiles:

Start specific services only when requested (e.g., debug tools).

Networking

Service Discovery (DNS)

Compose automatically sets up a network for your app. Services can reach each other by using their service name as the hostname.

Example: If your web service needs to talk to your database service named "db", the connection string is just postgres://user:pass@db:5432. No IPs needed!

Volumes & Persistence

Type Syntax Use Case
Named Volume services:
  db:
    volumes:
      - pg_data:/var/lib/... 		Databases. Data managed by Docker. Persists even if container is removed.
Bind Mount services:
  web:
    volumes:
      - ./src:/app/src 		Development. Live code changes reflect instantly in container.

Environment Variables

1. .env File (Default)

Compose automatically reads a .env file in the project root. 

# .env
POSTGRES_VERSION=13
API_KEY=12345

Usage in yaml: image: postgres:${POSTGRES_VERSION}

2. Environment Block

Inject vars directly into the container.

services:
  app:
    environment:
      - NODE_ENV=production
      - DB_PASS=${DB_PASS} # Pass-through
9

Scaling & Dependencies

$ docker compose up --scale worker=3 -d

This spins up 3 replicas of the 'worker' service.
⚠️ Warning: Do not bind host ports (e.g., 80:80) when scaling, or you'll get a "Port already allocated" error. Use a reverse proxy instead.

Logging & Observability

View All Logs docker compose logs -f

Aggregates output from all services into one stream.

Specific Service docker compose logs -f web

Focus on just the 'web' container.

11

Security Best Practices

Secrets Management

Never commit .env files containing production API keys or passwords to Git. Add .env to your .gitignore. Use Docker Secrets for sensitive data in production.

Least Privilege

Don't run containers as root. Use the user: "1000:1000" directive in common services like Node or Python.

Practical Labs

Lab 1: Local Development (Node + Postgres)

Typical Node.js API with a database. Uses host binding for easy access.

services:
  api:
    build: .
    ports:
      - "3000:3000"
    environment:
      - DB_URL=postgres://user:pass@db:5432/myapp
    depends_on:
      - db
    volumes:
      - ./src:/app/src  # Live Reload

  db:
    image: postgres:15-alpine
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: myapp
    ports:
      - "5432:5432" # Access DB from host tools like DBeaver

Lab 2: Full Stack (React + Python + Redis)

Complex stack with frontend, backend, and cache. 

services:
  frontend:
    build: ./frontend
    ports: ["80:80"] # Production build
    depends_on: [backend]

  backend:
    build: ./backend
    expose: ["8000"] # Only internal access
    depends_on: [redis]

  redis:
    image: redis:alpine
    volumes:
      - redis_data:/data

volumes:
  redis_data:

Lab 3: Reverse Proxy (NGINX)

Route path requests /api to backend and / to frontend.

services:
  proxy:
    image: nginx:alpine
    ports: ["80:80"]
    volumes:
      - ./nginx.conf:/etc/nginx/conf.d/default.conf:ro
    depends_on:
      - web
      - api

  web:
    image: my-web-app

  api:
    image: my-api-app

Lab 4: Stateful MongoDB Cluster

Running a database with persistent storage.

services:
  mongo:
    image: mongo:latest
    restart: always
    environment:
      MONGO_INITDB_ROOT_USERNAME: root
      MONGO_INITDB_ROOT_PASSWORD: example
    volumes:
      - mongo-data:/data/db

  mongo-express:
    image: mongo-express
    ports: ["8081:8081"]
    environment:
      ME_CONFIG_MONGODB_ADMINUSERNAME: root
      ME_CONFIG_MONGODB_ADMINPASSWORD: example
      ME_CONFIG_MONGODB_SERVER: mongo

volumes:
  mongo-data:

Lab 5: Multi-Env Profiles

Run extra tools only when needed (e.g., debug). 

services:
  web:
    image: myapp

  # Only runs with --profile debug
  # Command: docker compose --profile debug up
  phpmyadmin:
    image: phpmyadmin/phpmyadmin
    profiles: ["debug"]
    depends_on: [db]

  db:
    image: mysql

Compose Cheat Sheet

Category Command Description
Lifecycle docker compose up Create and start containers
Lifecycle docker compose up -d Start in detached mode (background)
Lifecycle docker compose down Stop and remove containers, networks
Lifecycle docker compose down -v Remove volumes as well (Dangerous!)
Lifecycle docker compose start Start existing containers
Lifecycle docker compose stop Stop active containers
Lifecycle docker compose restart Restart containers
Lifecycle docker compose build Build or rebuild services
Lifecycle docker compose pull Pull service images
Logs & Ops docker compose logs View output from services
Logs & Ops docker compose logs -f Follow log output
Logs & Ops docker compose ps List containers
Logs & Ops docker compose top Display the running processes
Logs & Ops docker compose images List images used
Logs & Ops docker compose port Print public port for a port binding
Execution docker compose exec Execute a command in a running container
Execution docker compose run Run a one-off command
Execution docker compose cp Copy files/folders between service and local
Execution docker compose config Validate and view the Compose file
