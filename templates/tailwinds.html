2. Getting Started

🧠 Mental Model

Tailwind v4 is a compiler, not just a CSS file. It scans your HTML, finds classes like flex, and generates the exact CSS needed. v4 uses a new Rust-based engine for instant builds.

What is it?

A utility-first CSS framework packed with classes like flex, pt-4, text-center.

Why it exists?

To stop you from fighting with CSS specificity, naming conventions, and "append-only" stylesheets that grow forever.

1. Core Concepts

1.1 Utility-First Fundamentals

🧠 Mental Model

Don't invent class names like .sidebar-inner-wrapper. Instead, describe the appearance directly in HTML using primitives. Think of it as "styling inline", but with constraints and a design system.

What is it?

Building complex components from a constrained set of primitive utilities.

Why it exists?

To stop you from writing the same CSS properties over and over again, and to prevent your CSS file from growing linearly with your HTML.

Real-life Use Case

Rapidly prototyping a card component without leaving your HTML file or worrying about naming collisions.

<div class="p-6 max-w-sm mx-auto bg-white rounded-xl shadow-lg flex items-center space-x-4">

1.2 Hover, Focus, and Other States

🧠 Mental Model

Every utility class in Tailwind can be applied conditionally by adding a modifier to the beginning of the class name.

What is it?

Modifiers like hover:, focus:, active:, disabled:, group-hover:.

Why it exists?

To keep all your state-based styling in your markup, rather than writing separate CSS blocks for each state.

<button class="bg-sky-500 hover:bg-sky-700 active:bg-sky-800 text-white font-bold py-2 px-4 rounded transition-colors">
  Save Changes
</button>

1.3 Responsive Design

🧠 Mental Model

Mobile First. Build the mobile layout first (default), then layer on changes for larger screens using breakpoints.

What is it?

Breakpoints: sm (640px), md (768px), lg (1024px), xl (1280px), 2xl (1536px).

Why it exists?

To create adaptive layouts without writing custom media queries.

<!-- Width is 16 by default, 32 on md screens, 48 on lg screens -->
<div class="w-16 md:w-32 lg:w-48 h-16 bg-pink-500 rounded-lg mx-auto transition-all duration-500"></div>

1.4 Dark Mode

🧠 Mental Model

Just another state variant. Use dark: to style elements when dark mode is enabled.

What is it?

A modifier that applies styles when the user's OS preference is dark or a class is present.

Real-life Use Case

Inverting text and background colors for night-time readability.

1.5 Reusing Styles

🧠 Mental Model

Don't rely on CSS classes for reuse. Rely on Components (React, Vue, Blade) or editor multi-cursor features.

What is it?

Extracting repeating utility patterns into template partials or components.

Why it exists?

To maintain the Single Source of Truth. If you need to change a button's style, you change it in the Button component.

<!-- Don't do this repeatedly -->
<button class="py-2 px-4 bg-blue-500 text-white rounded">Btn 1</button>
<button class="py-2 px-4 bg-blue-500 text-white rounded">Btn 2</button>

<!-- Do this (React Example) -->
function Button({ children }) {
  return (
    <button class="py-2 px-4 bg-blue-500 text-white rounded">{children}</button>
  );
}

1.6 Adding Custom Styles

🧠 Mental Model

Use Arbitrary Values for one-off values, and CSS Variables or Config for design system values.

Arbitrary Values

w-[32rem], bg-[#bada55]. Use square brackets to pass any CSS value.

Arbitrary Properties

[mask-type:luminance]. Use square brackets for CSS properties not in Tailwind.

<div class="bg-[#1da1f2] text-white p-[15px] rounded-[10px]">

1.7 Functions & Directives

🧠 Mental Model

Special syntax used in your CSS files to interact with Tailwind.

@theme

Defines your design system variables (colors, fonts, spacing).

@apply

Inlines existing utilities into custom CSS classes. Use sparingly.

@variant

Defines custom variants like hocus: (hover + focus).

@theme {
  --color-brand: #ff00ff;
}

.btn-primary {
  @apply py-2 px-4 bg-brand text-white rounded;
}

2.3 Theme Configuration

🧠 Mental Model

The `@theme` block is your design system's source of truth. It maps abstract names (like `primary`) to concrete values (like `#3b82f6`).

Extending

Values in `@theme` add to the defaults. To override, you'd use specific syntax (or just rely on precedence).
