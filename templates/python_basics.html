Module 1: Python Foundations

What is Python

💡 What Is It?

Python is a high-level, interpreted programming language. Code runs line-by-line through an interpreter, not compiled to machine code first. Known for readable syntax and "batteries included" philosophy.

🔧 Why It Exists?

Created to prioritize developer productivity and code readability over raw performance. Guido van Rossum wanted a language that was fun to use and easy to teach.

⚙️ How It Works?

CPython (standard Python) converts your .py files to bytecode (.pyc), then executes it in a virtual machine. Interpreter reads bytecode and executes instructions. JIT compilation available via PyPy for speed.

🎯 Where Used?
  • Web development (Django, Flask, FastAPI)
  • Data analysis (pandas, NumPy)
  • Machine learning (TensorFlow, PyTorch)
  • Automation scripts
  • API development
✅ When To Use?

Use Python for rapid prototyping, data processing, web backends, scripting, ML/AI, and when team productivity matters more than microsecond performance.

⚠️ When To Avoid?

Avoid for: mobile apps (use Swift/Kotlin), real-time systems requiring guaranteed latency, CPU-intensive tasks needing C++ speed, or when memory is extremely limited.

🚀 Modern Python with UV (The New Standard)

💡 What is UV?

UV is an extremely fast Python package and project manager written in Rust. It replaces pip, venv, poetry, and pyenv with a single, lightning-fast tool.

# macOS / Linux Installation

Hands-on Interactive Migration Guide (Pip ➔ UV)

Click on each task below to see the evolution from Legacy tools to full UV adoption. The intent is to migrate step-by-step until you only use native UV commands.

1. Set Up Python & Virtual Environment

🐢 Legacy

Requires Pyenv for version management, slow setup.

pyenv install 3.12 
python3.12 -m venv .venv
source .venv/bin/activate

🔄 Transition

Fast venv creation with UV, usual active workflow.

uv venv --python 3.12
source .venv/bin/activate

🚀 Native UV

No manual activation. UV handles it automatically.

# UV manages env automatically!
uv run --python 3.12 script.py

# Or initialize full project
uv init my_project
2. Install Dependencies (Single Package)

🐢 Legacy

Slow, resolves dependencies one by one.

# Active venv required
pip install requests

🔄 Transition

10-100x faster drop-in replacement for Pip.

# Active venv required
uv pip install requests

🚀 Native UV

Adds to config, runs instantly with project info.

# Adds to pyproject.toml
uv add requests

uv run script.py
3. Bulk Installs (requirements.txt)

🐢 Legacy

Installs from file. Hard to lock dependencies.

pip install -r reqs.txt
pip freeze > reqs.txt

🔄 Transition

Lightning fast bulk install, Pip-compatible.

uv pip install -r reqs.txt
uv pip compile > reqs.txt

🚀 Native UV

Deterministic sync with uv.lock file.

# Syncs with lockfile
uv sync

# Updates lockfile
uv lock
4. Install & Run Tools (Ruff, Pytest, FastApi)

🐢 Legacy

Requires system Pipx or clutters project venv.

pipx install ruff
ruff check .

🔄 Transition

Fast tool installation in isolated environments.

uv tool install ruff
ruff check .

🚀 Native UV

Run tools instantly with uvx WITHOUT installing!

# Downloads & runs on-the-fly!
uvx ruff check .
uvx pytest
# Or specific version:
uvx ruff@0.6.0 check .

⚡ The UV Project Model: Concepts & Anatomy
1. The Project Anatomy (The Files Created)

When you run uv init, it creates a standardized modern python project structure.

my-project/
├── pyproject.toml      # THE MANIFEST: Defines metadata and high-level dependencies.
├── uv.lock             # THE SOURCE OF TRUTH: Fully resolved, cross-platform dependency tree.
├── .python-version     # THE PIN: Tells uv exactly which Python version to use for this folder.
├── .venv/              # THE ENVIRONMENT: Local isolated folder where packages are linked.
├── .pdm-python         # (Opt) Cache file for some setups to track interpreter.
└── hello.py            # Boilerplate script.
2. Dependency Management Concepts
  • uv add <pkg>
    Adds name to config, resolves tree, updates lock, syncs .venv.
  • uv remove <pkg>
    Removes from config, re-resolves lock, prunes .venv.
  • uv sync
    IDEMPOTENT: Makes .venv EXACTLY match uv.lock.
  • uv lock
    Calculates the math of dependencies; does NOT touch .venv.
3. The Global Cache ("Magic" of Speed)

Content-Addressable Storage:

  • uv downloads a package ONLY ONCE to a global system folder.
  • Instead of copying files into .venv, it uses HARD LINKS (macOS/Linux) or Reflinks/Copies (Windows).
  • Result: 100 projects using pandas take disk space of ONLY ONE copy.
4. Python Interpreter Management

Managed Pythons:

  • uv does not rely on your "System Python" (the one Apple/Linux provides).
  • It fetches standalone, portable Python builds ('python-build-standalone').
  • You never have to install Python manually again; uv run will auto-fetch it if missing.
5. Configuration Hierarchy

uv looks for configuration in this order:

  1. CLI Flags: --python 3.12 or --offline
  2. Env Vars: UV_PYTHON_INSTALL=manual
  3. Project Config: [tool.uv] block in pyproject.toml
  4. User Config: ~/.config/uv/uv.toml (Global settings)
6. Transient vs Persistent Envs
  • uv run <file.py>
    Creates a hidden, ephemeral environment if one doesn't exist.
  • uv tool run <cmd>
    (uvx) Downloads the tool to a shared cache, runs it, then leaves it alone.
  • uv sync
    Manages the PERSISTENT .venv in your current working directory.
7. Workspaces (Monorepos)

Concept: One lockfile for multiple packages.

[tool.uv.workspace]
members = ["packages/*"]
  • Shared lockfile ensures all sub-projects use compatible versions.
  • Fast local development: sub-packages are linked, not installed via pip.

Variables

💡 What Is It?

Variable is a name that refers to a value in memory. In Python, variables are references (pointers) to objects, not containers holding values.

🔧 Why It Exists?

Variables let you store, reuse, and modify data. Without them, you'd have to hardcode every value. They make programs dynamic and maintainable.

⚙️ How It Works?

Assignment creates a name binding to an object. The object has a type, not the variable. Multiple names can refer to the same object. Python uses reference counting for memory management.

# Variable assignment
x = 10           # integer
name = "Bob"     # string
price = 19.99   # float
is_active = True # boolean

# Multiple assignment
a, b, c = 1, 2, 3

# Reassignment
x = "now a string"  # Dynamic typing

# Naming conventions
user_age = 30      # snake_case for variables
MAX_SIZE = 100    # UPPER_CASE for constants

🚫 Common Mistakes

  • Using reserved keywords as names (class, if, for)
  • Starting with numbers: 1var (invalid)
  • Assuming variable "contains" value (it's a reference!)
  • Not following naming conventions
🧠 Mental Model: Variable is a name tag attached to an object, not a box holding it. You can move the tag to point at different objects anytime.

Numbers and Math Operations

💡 What Is It?

Python has int (whole numbers), float (decimals), and complex numbers. Unlimited precision for integers.

# Standard ops

🧠 Mental Model: Integers are exact, floats are approximations. Think: int for counting, float for measuring.

Lists

💡 What Is It?

Ordered, mutable collection of items. Can hold mixed types. Dynamic size. Accessed by index (0-based).

🔧 Why It Exists?

Store multiple items in sequence. Maintain order. Modify contents. Most versatile data structure for general use.

⚙️ How It Works?

Implemented as dynamic array. Stores references to objects. Resizes automatically. O(1) append, O(n) insert at beginning.

Data Access
Lists are ordered and mutable (changeable). Access elements by index. 
# Index Access
lst = ["a", "b", "c"]
lst[0]      # "a"
lst[-1]     # "c"

# Iteration
for item in lst:
    print(item)

Mutability vs Immutability

💡 What Is It?

Mutable: can change (list, dict, set). Immutable: can't change (int, str, tuple). Critical concept!

# Immutable (string)
s = "hello"
# s[0] = 'H'  # Error!
s = s.upper()  # Creates NEW string

# Mutable (list)
lst = [1, 2, 3]
lst[0] = 99     # Modifies in place

# Aliasing with mutable
a = [1, 2]
b = a        # b points to same list!
b.append(3)
print(a)      # [1, 2, 3] - a changed too!

# Copy instead
c = a.copy()  # New list

🚫 Common Mistakes

  • Aliasing lists: b = a (use b = a.copy())
  • Default mutable args: def f(lst=[]) - dangerous!
  • Modifying list while iterating over it

Concept: Iterable vs Indexable

🔑 Key Idea: Loops vs Positions

Some objects can be looped over (Iterable), some can be accessed by position (Subscriptable/Indexable), and some do both.

Fundamentals:

  • Iterable (__iter__) → Supports for x in obj
  • Indexable (__getitem__) → Supports obj[i]
  • dict_keys → Iterable ✅ Indexable ❌

⚠️ Why this matters: Dictionaries are KEY-based, not POSITION-based. Python prevents index access on views (`.keys()`, `.values()`) to avoid misleading logic.

🧩 Views are Windows, Lists are Snapshots
dict.keys() returns a VIEW object, unrelated to lists. To index it, you must explicitly convert it. 
d = {"E101": "IT", "E102": "HR", "E103": "IT"}

keys_view = d.keys()

# Here is how the keys_view object looks like
print(keys_view)
# dict_keys(['E101', 'E102', 'E103'])

# ✅ Allowed: Iteration
for k in keys_view:
    print(k)

# ❌ Error: TypeError (not indexable)
# keys_view[-1]

# ✅ Explicit Conversion (Snapshot), as keys_view is not an list
keys_list = list(keys_view)
print(keys_list[-1])       # "E103"

Concept: Accessing Complex Objects

🧠 Mental Model: The "Russian Dolls" Approach

Complex data structures are just nested containers. To access data, start from the outside and peel back layers one by one.

Step-by-Step Untangling Method (Always Works):

  • Step 1: Identify Type. type(obj). Dict? List?
  • Step 2: Ask "How do I access?"
    • List (Sequence) → Index [0]
    • Dict (Map) → Key ["key"]
  • Step 3: Discover Content. print(obj.keys()) (for dicts) or len(obj) (for lists).
  • Step 4: Drill Down ONE Level. Assign to a temporary variable (e.g., level1 = data[0]) before going deeper.
🧩 Real Example: Navigating JSON-like Data
Imagine you receive this messy data from an API. You want Ravi's first role. 
data = {
    "users": [
        {"name": "Asha", "roles": ["admin", "editor"]},
        {"name": "Ravi", "roles": ["viewer"]}
    ]
}

# bad: data["users"][1]["roles"][0] (Hard to debug)

# Good: Drill down step-by-step
users_list = data["users"]      # 1. Dict Key -> List
ravi_dict  = users_list[1]      # 2. List Index -> Dict
ravi_roles = ravi_dict["roles"] # 3. Dict Key -> List
target     = ravi_roles[0]      # 4. List Index -> String

print(target)  # "viewer"
🛠️ Inspection Helper
When dealing with messy or unknown data (like from an API), use this recursive function to inspect the structure without drowning in text. 
def inspect(obj, indent=0):
    pad = "  " * indent
    if isinstance(obj, dict):
        for k, v in obj.items():
            print(f"{pad}{k}: {type(v).__name__}")
            inspect(v, indent+1)
    elif isinstance(obj, list):
        print(f"{pad}list[{len(obj)}]")
        if obj:
            inspect(obj[0], indent+1)

💡 What Is It?

💡 What Is It?

*args and **kwargs

💡 What Is It?

# *args - variable positional
def sum_all(*args):
    print(args)  # Tuple
    return sum(args)

print(sum_all(1, 2, 3))       # 6
print(sum_all(1, 2, 3, 4, 5))  # 15

# **kwargs - variable keyword
def print_info(**kwargs):
    print(kwargs)  # Dict
    for key, value in kwargs.items():
        print(f"{key}: {value}")

print_info(name="Alice", age=25, city="NYC")

# Combined
def func(a, b, *args, **kwargs):
    print(f"a={a}, b={b}")
    print(f"args={args}")
    print(f"kwargs={kwargs}")

func(1, 2, 3, 4, x=10, y=20)
When To Use

Wrapper functions, decorators, flexible APIs, logging functions.

File System Navigation (pathlib)

💡 What Is It?

Object-Oriented way to handle filesystem paths. Instead of treating paths as simple strings, pathlib treats them as objects with methods and properties.

🔧 Why It Exists?

To abstract away Operating System differences (Windows uses \, Mac/Linux use /). It eliminates messy string manipulation and makes code readable and cross-platform safe.

⚙️ How It Works?

It uses operator overloading (the / symbol) to join paths naturally. You create a Path object and chain methods to perform operations.

Key Methods & Operations (folder/directory/files)

Module 10: Object-Oriented Python(OOP)

💡 What Is It?

Object-Oriented Programming (OOP) is a programming paradigm based on the concept of "objects", which can contain data (attributes) and code (methods). It structures programs so that properties and behaviors are bundled into individual objects.

🔧 Why It Exists?

To manage complexity in large software systems. It promotes code reuse, modularity, and scalability by mimicking real-world entities and their interactions, making code easier to maintain and extend.

⚙️ How It Works?

You define Classes (blueprints) that specify the structure and behavior. You then create Objects (instances) from these classes. Key concepts include Encapsulation (hiding data), Inheritance (reusing code), and Polymorphism (flexible interfaces).

🎯 Where Used?
  • Large-scale applications (Enterprise software)
  • GUI development (PyQt, Tkinter)
  • Game development (Pygame, Unity/C#)
  • Web Frameworks (Django models)
  • Data modeling
✅ When To Use?

Use OOP when dealing with complex systems with many interacting parts, when you need to model real-world relationships, or when working on large teams where modularity and encapsulation are crucial for preventing bugs.

⚠️ When To Avoid?

Avoid for simple, small scripts where a procedural approach is faster to write and easier to read. Also, functional programming might be better for data transformation pipelines where state mutation is undesirable.

📚 Params vs Args vs Attrs
  • Parameter: Variable defined in function declaration (placeholder).
  • Argument: Actual value passed during function call.
  • Attribute: Data stored inside an object/instance (self.x). Attributes are variables that store data in a class associated with an object.
Classes & Objects
Class is a blueprint; Object is an instance. Combines data (attributes) and behavior (methods).
class Dog:

    def bark(self):

Project 1: File Organizer Script
Automatically organize downloads folder by file type. Move PDFs to Documents/, images to Pictures/, etc.
import os
import shutil

def organize_files(directory):
    """Organize files by extension into folders."""
    file_types = {
        'Documents': ['.pdf', '.docx', '.txt'],
        'Images': ['.jpg', '.png', '.gif'],
        'Videos': ['.mp4', '.avi'],
        'Code': ['.py', '.js', '.html']
    }
    
    for filename in os.listdir(directory):
        if os.isfile(os.path.join(directory, filename)):
            ext = os.path.splitext(filename)[1]
            
            for folder, extensions in file_types.items():
                if ext in extensions:
                    dest_folder = os.path.join(directory, folder)
                    os.makedirs(dest_folder, exist_ok=True)
                    
                    src = os.path.join(directory, filename)
                    dst = os.path.join(dest_folder, filename)
                    shutil.move(src, dst)
                    print(f"Moved {filename} to {folder}/")

organize_files("/path/to/downloads")
